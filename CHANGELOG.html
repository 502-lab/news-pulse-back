Bug Fix HIGH cors.allowed-origins YAML 리스트 @Value 주입 실패 → @ConfigurationProperties 전환
▼
문제
모든 @SpringBootTest 통합 테스트가 PlaceholderResolutionException: Could not resolve placeholder 'cors.allowed-origins'로 컨텍스트 로드 실패.
근본 원인
YAML 리스트(- item 형식)는 Spring Environment에 cors.allowed-origins[0], cors.allowed-origins[1] 등의 indexed key로 저장됨. @Value("${cors.allowed-origins}")의 Placeholder resolver는 해당 prefix만으로 단일 값을 찾으므로 해소 불가. Spring Boot 4.x / Spring Framework 7.x에서 이 동작이 엄격해짐.
수정 내용
CorsProperties record (@ConfigurationProperties(prefix = "cors")) 신규 생성. SecurityConfig에서 @Value 필드 제거하고 생성자 주입으로 전환. NewsCuratorApplication에 @EnableConfigurationProperties(CorsProperties.class) 추가.
결정 이유
@ConfigurationProperties는 YAML 리스트를 List<String>으로 직접 바인딩하는 Spring Boot 공식 방식. 대안인 YAML을 콤마 구분 문자열로 변경하면 가독성이 저하되고, 각 Origin에 포트·도메인 변경 시 오타 위험 증가.

근본 원인
JwtTokenProvider.generatePendingSignupToken()이 userInfo(Apple name JSON)를 JWT claim에 포함. Redis 등 서버 사이드 스토리지 없이 stateless 설계를 유지하면서도 PII를 최소화하려면 클라이언트가 /complete 시점에 재전송하는 방식이 적합. 테스트의 직접 JWT 생성은 /authorize 서버 로직 변경 시 테스트가 이를 감지하지 못하는 구조적 약점.

  • JwtTokenProvider.generatePendingSignupToken(): userInfo 파라미터 및 claim 제거
  • SocialCompleteRequest: nullable userInfo 필드 추가
  • SocialAuthService.complete(): userInfo 파라미터 추가, JWT claim 대신 요청 바디에서 수신
  • SocialAuthController.complete(): request.userInfo()를 서비스에 전달
  • SocialAuthService.callback(): storedUserInfo 변수 제거, generatePendingSignupToken 3-arg 호출

결정 이유
Redis 추가 없이 PII 노출 최소화. email은 계정 생성에 필수이므로 JWT에 유지하고, userInfo(Apple name)는 순수 표시 데이터이므로 클라이언트 재전송 방식 채택. 악의적 클라이언트가 다른 userInfo를 보내도 인증 결정에 영향 없음. 테스트의 /authorize 경유 방식은 실제 엔드포인트와 state 생성 로직을 공유해 회귀를 더 잘 감지.

  • Hibernate 6 Instant 타입 오류 — Spring Boot 4 / Hibernate 6에서 PostgreSQL TIMESTAMPTZ 컬럼을 네이티브 쿼리 Object[]로 받으면 java.sql.Timestamp 대신 java.time.Instant이 반환됨. toOffsetDateTime() 헬퍼에 instanceof Instant 분기 추가로 근본 수정.
