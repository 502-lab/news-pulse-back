Hotfix Bug Fix CRITICAL 수집 파이프라인 전체 중단 — @Modifying + RETURNING 쿼리 예외
스모크 테스트에서 발견
▼
문제

연합뉴스·네이버-경제 출처의 consecutive_failure_count가 매 실행마다 증가하며 기사가 1건도 저장되지 않음. RSS/API 자체는 정상 응답하는데 collectFromSource가 예외로 종료.

근본 원인

Hibernate 6는 @Modifying 존재 시 executeUpdate() 경로를 강제. PostgreSQL RETURNING은 결과 셋을 반환하므로 JDBC가 "A result was returned when none was expected" 예외 발생 → @Transactional 롤백 → 저장 0건, failure_count 증가.

수정
  • incrementAndGetCallCount 단일 쿼리를 upsert(void) + select(int) 두 쿼리로 분리
  • upsertCallCount: @Modifying(flushAutomatically=true), RETURNING 없는 upsert
  • selectCallCount: 별도 native SELECT로 실제 카운트 반환
결정 이유
  • @Modifying 제거 후 SELECT 단일 쿼리로 변경은 flush 타이밍 보장이 없어 stale 카운트 반환 가능 → 기각
  • EntityManager 직접 사용은 Spring Data JPA 추상화를 포기하고 보일러플레이트 증가 → 기각
  • 두 쿼리 분리는 변경 범위 최소화 + Spring Data JPA 계약 유지 + flushAutomatically로 동일 트랜잭션 내 정합성 보장. 가장 작은 수정으로 문제 해결.

Bug Fix HIGH 한겨레 RSS 수집 실패 — Java User-Agent 차단 (403)
스모크 테스트에서 발견
▼
문제

RestClient가 기본 User-Agent(Java/25.0.2)로 요청 시 한겨레 nginx가 차단. RssSourceAdapter catch에서 emptyList() 반환 → 기사 0건.

근본 원인

Spring RestClient의 기본 UA가 Java/{버전}. 한겨레는 이를 봇으로 판단해 403 반환.

수정

RestClientConfig에 defaultHeader("User-Agent", "Mozilla/5.0 (compatible; NewsCurator/1.0; ...)") 추가.

결정 이유
  • 어댑터별(소스별) UA 설정은 새 소스 추가 시 누락 가능, 일관성 유지 어려움 → 기각
  • RestClient 전역 default header로 설정하면 모든 소스가 동일한 UA를 사용. 향후 다른 소스도 Java UA를 차단할 경우 한 곳에서만 수정하면 됨. 관리 포인트 최소화.

▼
문제

GET /api/v1/articles?size=-1 → HTTP 200 반환. FeedRequest record의 @Positive는 있으나 Controller가 new FeedRequest(cursor, size, categoryEnum)를 직접 생성해 @Valid가 동작하지 않음. clampSize()가 음수를 기본값으로 처리.

수정

문제

근본 원인

  • implementation= 사용: SpringDoc이 해당 클래스를 직접 파싱해 인라인 스키마를 생성하므로 별도 등록 불필요. 사용 지점에서 의도가 명확하게 드러남.

BUG FIX CRITICAL CollectionService.collectAll() self-invocation으로 @Transactional 무효화 → 수집 0건
service 분리
▼
문제 (발견 경위)
collectAll() 내부에서 this.collectFromSource(source)를 직접 호출. Spring AOP 프록시를 우회하므로 collectFromSource의 @Transactional이 전혀 적용되지 않아 articles 수집 결과가 0건.
근본 원인
Spring @Transactional은 프록시 기반 AOP로 동작. 동일 빈 내부에서 this.method() 호출 시 프록시를 거치지 않아 어드바이스(트랜잭션 경계)가 무시됨.
수정 내용
collectFromSource 및 관련 private 헬퍼(processCandidate, findAdapter)를 SourceCollectionExecutor 빈으로 분리. CollectionService는 순수 오케스트레이터로 단순화하여 sourceCollectionExecutor.collectFromSource(source)를 호출. CollectionServiceTest는 collectAll 오케스트레이션만 검증하도록 리팩터링, collectFromSource 케이스는 SourceCollectionExecutorTest로 이동.
결정 이유
대안①: @Lazy 자기 주입 — 간단하나 순환 의존성이 발생하고 코드 의도가 불명확. 대안②: AopContext.currentProxy() — AspectJ 위빙 설정 필요, 빌드 복잡도 증가. 선택안: 별도 빈 분리는 단일 책임 원칙에 부합하고, 테스트 시 두 클래스를 독립적으로 mock할 수 있어 구조적으로 가장 깔끔.

Bug Fix HIGH 한겨레 RSS 308 Permanent Redirect 미처리 → RSS 본문 대신 /rss 4바이트 반환
수집 파이프라인 검증에서 발견
▼
문제 (발견 경위)

https://www.hani.co.kr/rss/ 수집 트리거 시 ROME이 SyndFeedException: Invalid XML: Error on line 1: 프롤로그에서는 콘텐츠가 허용되지 않습니다를 던지며 한겨레 수집 0건. DB에는 consecutive_failure_count 증가 없이 조용히 실패.

근본 원인

서버가 해당 URL을 HTTP 308로 /rss(4바이트)로 영구 이전함. SimpleClientHttpRequestFactory(내부 HttpURLConnection)는 301/302/303만 자동 follow하고 307/308은 follow하지 않아 응답 body로 /rss 텍스트가 그대로 반환됨. ROME이 이를 XML로 파싱 시도 → 예외.

수정 내용

RestClientConfig에서 SimpleClientHttpRequestFactory를 JdkClientHttpRequestFactory(HttpClient.Redirect.NORMAL)로 교체. NORMAL 정책은 301/302/303/307/308 모두 follow하며 HTTPS→HTTP 다운그레이드만 차단.

결정 이유
  • DB seed URL을 /rss로 변경: 현재 URL은 영구 redirect이므로 단기적으로는 동작하나, redirect 체인이 다시 변경될 경우 DB를 재수정해야 함. 또한 CLAUDE.md에 "시드 URL 직접 수정 금지" 원칙에 위배. 기각.
  • 어댑터에서 수동 redirect follow 구현: redirect 처리 코드가 HTTP 클라이언트 레이어가 아닌 비즈니스 레이어에 노출됨. 관심사 분리 위반. 기각.
  • JdkClientHttpRequestFactory 교체: Java 표준 HttpClient의 redirect 정책을 이용하므로 별도 코드 없음. 모든 RSS/Naver 출처의 307/308 redirect 대응이 전역으로 적용됨. RestClientConfig 한 곳만 수정하므로 최소 변경.

Bug Fix HIGH RSS/Naver 페치·파싱 실패를 성공(0건)으로 오기록 → consecutive_failure_count 증가 안 함
수집 파이프라인 검증에서 발견
▼
문제 (발견 경위)

RssSourceAdapter·NaverSourceAdapter의 catch 블록이 HTTP/파싱 예외를 Collections.emptyList()로 삼켜 반환. SourceCollectionExecutor는 예외 없이 recordCollected()를 호출 → consecutive_failure_count가 0 유지. 운영자가 출처 장애를 인지할 방법 없음.

근본 원인

어댑터 catch 블록 설계 실수: 하드 실패(HTTP 오류·파싱 예외)와 소프트 빈 결과(정상 응답이지만 기사 0건)를 동일하게 처리. 하드 실패도 빈 리스트 반환으로 마스킹됨.

수정 내용
  • RssSourceAdapter: catch 블록에서 return emptyList() → throw new RuntimeException(..., e). null/빈 body에 대한 soft empty는 유지.
  • NaverSourceAdapter: 동일 패턴 수정.
  • 예외 전파 시 CollectionService.collectAll()가 recordFailure() 호출 → consecutive_failure_count 증가, ERROR 로그 기록.
결정 이유
  • emptyList() 유지 + 별도 모니터링 메트릭: 예외 마스킹 구조 그대로 두고 지표만 추가하면 근본 원인이 해결되지 않음. 기각.
  • 어댑터에서 직접 recordFailure() 호출: 어댑터는 HTTP 클라이언트 레이어로 DB Repository 의존성을 가져선 안 됨. 계층 경계 위반. 기각.
  • 예외 전파 + CollectionService에서 처리: CollectionService가 이미 catch → recordFailure() 구조를 갖추고 있음. 어댑터는 예외를 그대로 전파하기만 하면 되므로 최소 변경으로 올바른 책임 분리 유지.

Bug Fix MEDIUM Gemini 429 일시 오류를 영구 FAILED와 동일하게 처리 — AiTransientException 분리
AI 처리 파이프라인 런타임 검증에서 발견
▼
문제 (발견 경위)

Gemini 무료 티어 15 RPM 한도 초과 시 429 응답. 기존 callGeminiWithRetry는 내부적으로 최대 3회 재시도했지만 실패 시 AiProviderException으로 래핑 → processCategory/processSummary가 retry_count를 소모 → 90건이 FAILED 상태로 영구 고착.

근본 원인

429(rate limit), 5xx(서버 오류), 연결 타임아웃 등 재시도 가능한 일시적 오류와 400(잘못된 요청) 등 영구 오류를 동일한 예외 타입(AiProviderException)으로 처리하여 서비스 레이어에서 구분 불가.

수정 내용
  • AiTransientException extends AiProviderException 신규 생성.
  • GeminiAiProvider.callGemini(): HttpClientErrorException.TooManyRequests(429) → AiTransientException; HttpServerErrorException(5xx) → AiTransientException; ResourceAccessException(연결/타임아웃) → AiTransientException; 기타 4xx → AiProviderException(하드 실패).
  • callGeminiWithRetry 제거 — 배치 레벨에서 중단하는 방식으로 교체.
결정 이유
  • 내부 재시도 유지: 429 직후 1~4초 대기 후 재시도하면 동일 RPM 창 안에서 또 429를 맞음. 배치 전체를 더 사용하고 같은 결과. 기각.
  • 배치 중단 + 다음 스케줄 사이클: 스케줄 간격(60s)이 RPM 창보다 길어 자연스럽게 회복됨. 구현도 단순. 채택.
