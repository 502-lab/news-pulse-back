# Feature Specification: 010 인사이트 + 추천 — 개인 소비 리포트 & 놓친 기사 추천

**Feature Branch**: `feat/010-insights`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "010-insights 시작. 스코프 = 009 P1 + 006/003/007 기존 데이터 위 온디맨드 집계. 신규 테이블 0. P1 개인 소비 리포트 6항목 + P2 룰베이스 놓친 기사 추천. 평균읽기시간(P2 dwell 의존)·임베딩·사전집계 배치 제외."

## 개요

사용자가 자신의 뉴스 소비를 돌아볼 수 있는 **개인 소비 리포트**(읽은수·북마크·최다 카테고리·관심 분포·주요 언론사·내 편향)와, 관심사에 맞는데 아직 안 본 **놓친 기사 추천**을 제공한다. 모든 지표는 기존 데이터(009 조회 이력·003 북마크·006 편향·007 키워드/트렌드) 위에서 **온디맨드 집계**로 산출하며 **신규 테이블을 만들지 않는다**(MVP). 추천은 임베딩/ML 없이 **룰베이스**로 시작하되 교체 가능한 인터페이스 뒤로 격리한다. 클라이언트 계측이 필요한 **평균읽기시간(체류)**은 read-tracking P2 의존이라 본 스펙에서 제외한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 개인 소비 리포트 (Priority: P1)

로그인 사용자가 "내 인사이트"를 열면, 지금까지의 뉴스 소비를 요약한 리포트를 본다: 읽은 기사 수, 저장(북마크) 수, 가장 많이 읽은 카테고리, 관심 분포(카테고리/키워드 비중), 자주 읽은 언론사, 그리고 내가 읽은 기사들의 편향 분포(중립적 기술).

**Why this priority**: 009(조회 이력)·003·006·007이 쌓아온 데이터를 사용자에게 처음으로 "되돌려주는" 핵심 가치. 신규 테이블·외부 의존 없이 기존 데이터 집계만으로 즉시 가치를 만든다.

**Independent Test**: 한 사용자가 여러 기사를 조회·저장한 상태에서 `/api/v1/me/insights` 호출 → 6개 지표가 본인 데이터 기준으로 정확히 집계되어 반환되고, 타인 데이터는 섞이지 않음을 확인. 데이터가 없는 신규 사용자는 빈/0 값으로 안전하게 응답.

**Acceptance Scenarios**:

1. **Given** 사용자가 서로 다른 기사 N건을 조회, **When** 인사이트 조회, **Then** 읽은수=고유 기사 N(009 distinct)이 반환된다.
2. **Given** 사용자가 기사 M건을 북마크, **When** 인사이트 조회, **Then** 북마크 수=M이 반환된다.
3. **Given** 사용자가 여러 카테고리 기사를 조회, **When** 인사이트 조회, **Then** 최다 카테고리와 카테고리/키워드 관심 분포(비중)가 반환된다.
4. **Given** 사용자가 여러 언론사 기사를 조회, **When** 인사이트 조회, **Then** 자주 읽은 주요 언론사 목록이 반환된다.
5. **Given** 사용자가 편향 분석된 기사들을 조회, **When** 인사이트 조회, **Then** 읽은 기사들의 편향 분포가 **중립적 기술**(성향 단정 라벨 아님)로 반환된다(D1).
6. **Given** 조회/저장 이력이 거의 없는 신규 사용자, **When** 인사이트 조회, **Then** 오류 없이 빈/0 값 또는 최소표본 안내로 응답한다(D3).
7. **Given** 사용자 A 토큰, **When** 인사이트 조회, **Then** A 본인 데이터만 집계되고 타인 데이터는 0건 반영된다.

### User Story 2 - 놓친 기사 추천 (Priority: P2)

사용자가 "놓친 기사"를 열면, 관심사에 부합하지만 아직 보지 않은(저장도 안 한) 최근/주목 기사를 추천받는다.

**Why this priority**: 리포트(US1)가 과거 소비를 보여준다면, 추천은 미래 소비를 유도하는 실행 가치. 단 기존 스코어링·관심사·조회 이력 위에서 동작하므로 US1 다음 우선순위.

**Independent Test**: 관심사가 설정된 사용자가 일부 기사를 조회·저장한 상태에서 추천 호출 → 이미 본/저장한 기사는 제외되고, 관심사·최근성·트렌드 가중으로 정렬된 미열람 기사 N건이 반환됨을 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 일부 기사를 조회·저장, **When** 추천 조회, **Then** 이미 조회(009)·저장(003)한 기사는 추천에서 제외된다.
2. **Given** 사용자 관심사(카테고리/키워드)가 설정됨, **When** 추천 조회, **Then** 관심사 매칭 + 최근성 + 트렌드(007) 가중으로 정렬된 미열람 기사가 반환된다.
3. **Given** 조회 이력·관심사 둘 다 없는 콜드스타트 사용자, **When** 추천 조회, **Then** 트렌드/최근 인기 기사 fallback 추천이 반환된다(빈 응답 아님, D2).

---

### User Story 3 - [후속 사이클 — 본 스펙 제외] 심화 인사이트·추천 (Priority: P3)

평균읽기시간(체류·완료율 — read-tracking P2 dwell 의존), 임베딩 기반 추천, 인사이트 사전집계(주간 스냅샷 배치)는 본 스펙에서 구현하지 않으며 후속 사이클에서 다룬다.

**Why this priority**: 평균읽기시간은 클라이언트 계측 데이터(체류) 부재로 산출 불가(`article_event.metric_value`는 P1에서 항상 null). 임베딩 추천·사전집계는 인프라/볼륨 선행 필요. forward-seam만 유지.

**Independent Test**: 해당 없음(본 스펙 미구현).

---

### Edge Cases

- 읽은/저장 이력 0인 신규 사용자: 카운트는 0, 통계 항목(관심분포·내편향)은 "표본 부족" 플래그(읽은 고유 기사 < 5, D3). NPE·분모0 금지.
- 편향 분석 미완료(006 status≠DONE) 기사만 읽은 경우: 편향 분포는 분석 완료분만 집계(미완료 제외).
- 추천 후보가 부족(관심사 매칭 기사 소수): fallback(최근성/트렌드)으로 채우되 이미 본/저장 제외 유지.
- 숨김(008 admin_hidden) 기사: 리포트·추천 모두에서 제외(사용자향 노출 금지).
- 온디맨드 집계 비용: per-user GROUP BY 다수 → 응답 시간 한도 내(D4·SC-002).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자의 **읽은수(고유 기사 수, 009 조회 이벤트 distinct)**를 집계해 제공해야 한다.
- **FR-002**: 시스템은 사용자의 **북마크(저장) 수**(003 saved_articles)를 제공해야 한다.
- **FR-003**: 시스템은 사용자가 읽은 기사들의 **최다 카테고리**와 카테고리별 분포를 제공해야 한다.
- **FR-004**: 시스템은 사용자의 **관심 분포**(읽은 기사의 카테고리/키워드 비중, 007 키워드 활용)를 제공해야 한다.
- **FR-005**: 시스템은 사용자가 자주 읽은 **주요 언론사** 목록을 제공해야 한다(sources/article_sources).
- **FR-006**: 시스템은 사용자가 읽은 기사들의 **편향 분포**(006 분석 완료분)를 **중립적 기술**(예: 진보/중립/보수 비율 %, "당신은 ~성향" 단정 라벨 금지, D1)로 제공해야 한다. 표본 부족(읽은 고유 기사 < 5) 시 분포를 산출하지 않고 "표본 부족" 플래그로 응답한다(D3).
- **FR-007**: 인사이트·추천은 **인증된 본인만** 접근 가능해야 한다(타인 데이터 접근 불가, `/api/v1/me/...`).
- **FR-008**: 모든 리포트 지표는 **온디맨드 집계**로 산출하며 **신규 영속 테이블을 만들지 않는다**(사전집계 배치는 forward-note).
- **FR-009**: **데이터 부족 사용자**(읽은/저장 이력 0~소수)에 대해 오류 없이(NPE·분모0 없음) 빈/0 값으로 응답하고, **읽은 고유 기사 5개 미만이면 통계 항목(관심분포·내편향)에 "표본 부족" 플래그**를 표시해야 한다(분포 미산출, D3). 단순 카운트(읽은수·북마크수)는 항상 반환.
- **FR-010**: 시스템은 **놓친 기사 추천**을 제공해야 한다 — 후보 = **최근 14일 비숨김 기사**, **제외 = 이미 조회(009 article_event)·저장(003 saved_articles)·숨김(008 admin_hidden_at)**, **정렬 = 관심 카테고리 매칭 + 트렌드(007) + 최근성 가중**(미열람 기사 N건).
- **FR-011**: 추천 로직은 **교체 가능한 인터페이스 뒤로 격리**해야 한다(룰베이스 MVP, 임베딩 v2 교체 seam — 007 IssueClusterer 패턴 준용).
- **FR-012**: 추천 랭킹 초기 가중치는 **카테고리 0.5 / 트렌드 0.3 / 최근성 0.2**(plan에서 튜닝)를 따른다. **★ 콜드스타트**(조회이력·관심사 둘 다 없는 신규 사용자)는 **트렌드/최근 인기 기사 fallback**으로 빈 목록을 반환하지 않아야 한다(D2).

### Key Entities *(include if feature involves data)*

신규 영속 엔티티/테이블 **없음**(MVP — 온디맨드 집계). 아래는 응답 표현(read-only projection)일 뿐 저장하지 않는다.

- **InsightReport (응답 표현, 비영속)**: 읽은수·북마크수·최다카테고리·관심분포(카테고리/키워드 비중)·주요언론사·편향분포. 소스: 009 `article_event`·003 `saved_articles`·001 `articles/sources/article_sources`·006 `bias_analysis`·007 `article_keyword` 집계.
- **RecommendedArticle (응답 표현, 비영속)**: 추천 기사 + 추천 사유(관심사 매칭/트렌드/최근). 소스: `articles` 후보에서 조회·저장 제외 + 관심사·트렌드 스코어링.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 인사이트 6개 지표가 **본인 데이터 기준으로 정확**(읽은수=009 distinct, 북마크=003 count 등)하며, 타인 데이터는 **0건** 반영된다.
- **SC-002**: 인사이트·추천 조회가 **사용자 체감 즉시성**(저볼륨 per-user 집계 — 수백 ms 이내 응답 목표) 안에서 반환된다.
- **SC-003**: 추천 결과에 **이미 조회·저장한 기사가 0건** 포함된다(미열람·미저장 보장).
- **SC-004**: 데이터 부족 사용자도 **오류 없이**(NPE·분모0 없음) 인사이트/추천을 받는다 — 읽은 고유 기사 < 5면 통계 항목은 "표본 부족" 플래그, 추천 콜드스타트는 트렌드 fallback으로 비어있지 않다.
- **SC-005**: 편향 분포는 **중립적 기술**로 제공되어 성향 단정 라벨이 없다(D1 정합).

## Resolved Decisions (clarify 2026-06-24)

- **D1 — "내 편향" 표현**: **확정** — 중립적 기술(편향 분포 %, 단정 라벨 금지). 표본 부족 시 미표시. → FR-006.
- **D2 — 추천 룰**: **확정** — 후보 최근 14일·비숨김 / 제외 조회·저장·숨김 / 가중 category 0.5·trend 0.3·recency 0.2 / 콜드스타트 트렌드 fallback. → FR-010·FR-012.
- **D3 — 데이터 부족**: **확정** — 빈/0 안전 + 읽은 고유 기사 < 5면 통계 항목 "표본 부족" 플래그(분포 미산출). 카운트는 항상 반환. → FR-009.
- **D4 — 온디맨드 집계 성능**: **plan 이연** — per-user GROUP BY(저볼륨 가정), 인덱스 활용·쿼리 수 상한은 plan에서 구체화. SC-002 정합.

## Assumptions

- 모든 지표는 **읽은 기사(009 article_event VIEW)** 또는 **저장(003)** 을 기준으로 한 본인 스코프 집계다. 비로그인은 대상 아님(009와 동일).
- 편향 분포는 006 `bias_analysis.value`(분석 완료분)만 집계한다(미완료/실패 제외).
- 추천은 기존 `FeedService` 스코어링(categoryMatchScore 등)과 관심사(`user_interests`/`follow_keywords`)를 재활용한다.
- 숨김(008 admin_hidden) 기사는 리포트·추천에서 제외한다.
- 클라이언트 렌더링/UI는 범위 밖이다.

## Dependencies

- **009 (읽기 추적)**: `article_event`(VIEW) — 읽은수·최다카테고리·관심분포·주요언론사·내편향의 "읽은 기사" 기준.
- **003 (피드·검색·저장)**: `saved_articles`(북마크) + `FeedService` 스코어링 재활용(추천).
- **006 (편향 분석)**: `bias_analysis.value`(내 편향 분포).
- **007 (트렌드 집계)**: `article_keyword`(관심 키워드 분포) + 트렌드(추천 가중).
- **002 (인증)**: 본인 스코프 접근 제어, `user_interests`/`follow_keywords`.
