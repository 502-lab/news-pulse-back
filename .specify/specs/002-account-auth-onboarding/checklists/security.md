# Security Requirements Quality Checklist: 계정·인증·온보딩·인가

**Purpose**: 구현 시작 전 작성자 자기 검토 — 보안·위협 커버리지 관점에서 요구사항이 완전하고 명확하게 명세됐는지 검증
**Created**: 2026-06-11
**Feature**: [spec.md](../spec.md)
**Focus**: 보안 & 위협 커버리지 (토큰 보안, 브루트포스, RBAC, 소셜 인증, 이메일 인증)
**Audience**: 구현 담당자 자기 검토 (구현 착수 전)

> **읽는 법**: 각 항목은 요구사항 문서 자체의 품질(완전성·명확성·일관성)을 검사합니다.
> 구현 동작을 검증하는 것이 아니라, 요구사항이 잘 작성됐는지를 단위 테스트합니다.
> `[Gap]` = 요구사항 누락, `[Ambiguity]` = 모호한 표현, `[Conflict]` = 상충, `[Assumption]` = 명시 필요 가정

---

## ⛔ [REQUIRED GATE] RBAC 및 관리자 접근 제어

> 아래 5개 항목은 필수 통과 기준입니다. 미통과 시 구현 착수 금지.
> 배경: spec 001에서 `/admin/*`가 인증 없이 노출된 이슈 해소가 이 사이클의 핵심 목적 [Spec §FR-016, §US4]

- [ ] CHK001 - `/admin/*` 와일드카드가 커버하는 하위 경로(sub-path)와 허용 오퍼레이션이 spec에 명시됐는가, 아니면 와일드카드 범위만 암시적으로 처리됐는가? [Clarity, Spec §FR-016]
- [ ] CHK002 - 미인증(401)과 권한 부족(403)의 구분 기준이 보호된 모든 엔드포인트 유형에 대해 명확히 명세됐는가? [Clarity, Spec §US4]
- [ ] CHK003 - ADMIN 시드 계정 생성 방법(DB 시드 SQL vs 환경변수)의 보안 요건(시드 스크립트 접근 제한, 초기 자격증명 교체 등)이 요구사항에 포함됐는가? [Completeness, Spec §FR-017]
- [ ] CHK004 - ADMIN 토큰이 만료되거나 세션이 무효화된 이후 `/admin/*` 재접근 시 동작이 명세됐는가? [Gap, Spec §FR-016]
- [ ] CHK005 - 비밀번호 변경·재설정 후 ADMIN 세션도 강제 무효화 대상인지 명세됐는가? [Gap, Spec §FR-009, §FR-016]

## 토큰 보안 요구사항

- [ ] CHK006 - Refresh Token Rotation의 트리거 조건이 "사용 시마다 교체"로 명확히 명세됐는가, 아니면 특정 조건에서만 교체되는 가능성이 열려 있는가? [Clarity, Spec §FR-020]
- [ ] CHK007 - "무효화된 토큰 재사용 감지" 시나리오에서 정상적인 동시 요청(네트워크 지연으로 이전 토큰 재전송)과 실제 탈취를 구별하는 요건이 명세됐는가? [Ambiguity, Spec §FR-020]
- [ ] CHK008 - "계정 전체 세션 무효화" 범위가 "모든 기기의 모든 활성 RefreshToken"임이 명확히 명세됐는가? [Clarity, Spec §FR-020]
- [ ] CHK009 - 액세스 토큰(1시간)·리프레시 토큰(30일) 유효 기간이 Assumptions가 아닌 Functional Requirements 또는 Success Criteria에 명시됐는가? [Completeness, Spec §Assumptions]
- [ ] CHK010 - 비밀번호 변경 성공 후 해당 계정의 기존 액세스 토큰·리프레시 토큰 전체를 무효화하는 요건이 명세됐는가? [Gap]
- [ ] CHK011 - 리프레시 토큰의 기기 식별자(선택)가 보안 목적으로 사용되는지, 단순 UX 목적인지 요구사항에 명시됐는가? [Ambiguity, Spec §Key Entities]
- [ ] CHK012 - 리프레시 토큰 패밀리(토큰 발급 체인) 추적 없이 단순 해시 비교만으로 재사용을 감지할 수 있는지의 설계 전제가 요구사항에 반영됐는가? [Assumption, Spec §FR-020]

---

## 브루트포스 & 계정 잠금 요구사항

- [ ] CHK013 - 로그인 실패 카운터의 초기화 조건이 명세됐는가 — 성공적 로그인 시 초기화되는지, 잠금 해제 후 자동 초기화되는지, 아니면 두 경우 모두인지? [Clarity, Spec §FR-019]
- [ ] CHK014 - 계정 잠금 상태에서 추가 실패 시도가 잠금 기간(30분)을 연장하는지, 고정 유지인지 명세됐는가? [Ambiguity, Spec §FR-019]
- [ ] CHK015 - 로그인 실패 응답이 "계정 잠금"과 "잘못된 비밀번호"를 공격자에게 구별 가능하게 노출하는지, 아니면 동일 메시지로 반환하는지 명세됐는가? (계정 열거 방지) [Gap, Spec §FR-019]
- [ ] CHK016 - 계정 잠금 발생 시 계정 소유자(이메일)에게 알림을 보내는 요건이 필요한지 명세됐는가? [Gap, Spec §FR-019]
- [ ] CHK017 - 실패 횟수 집계 단위가 "계정 기준"으로만 명세됐는데, IP·기기 기반 보조 rate limiting 부재가 의식적 결정임을 요구사항에 명시하고 있는가? [Assumption, Spec §FR-019]

---

## 비밀번호 재설정 보안 요구사항

- [ ] CHK018 - 비밀번호 재설정 코드(6자리 숫자)의 엔트로피(10^6 조합)가 오입력 5회 제한과 결합해 충분한 보안 강도를 제공하는지 요구사항에 명시됐는가? [Completeness, Spec §FR-010, §FR-012]
- [ ] CHK019 - "코드 검증 성공 후 발급하는 변경 허가 토큰"이 단일 사용(single-use)임이 명세됐는가? [Clarity, Spec §US5]
- [ ] CHK020 - 비밀번호 재설정 성공 후 해당 계정의 모든 활성 세션을 강제 무효화하는 요건이 명세됐는가? [Gap]
- [ ] CHK021 - 비밀번호 재설정 엔드포인트가 등록된 이메일과 미등록 이메일에 동일한 응답을 반환해 계정 열거를 방지해야 한다는 요건이 명세됐는가? [Gap, Spec §US5]
- [ ] CHK022 - 이미 계정이 잠금 상태일 때 비밀번호 재설정 코드 발급 요청의 처리 방식이 명세됐는가? [Gap, Spec §Edge Cases]

---

## 이메일 인증 보안 요구사항

- [ ] CHK023 - "미인증" 상태에서 접근이 제한되는 기능 목록이 구체적으로 명세됐는가, 아니면 "보호된 기능"이라는 추상적 표현만 있는가? [Clarity, Spec §FR-022]
- [ ] CHK024 - 이메일 인증 코드 재발송 시 이전에 발급된 코드가 즉시 무효화되는지 명세됐는가? [Gap, Spec §FR-023]
- [ ] CHK025 - 이메일 인증 엔드포인트가 등록된 이메일과 미등록 이메일에 동일한 응답을 반환해 계정 열거를 방지해야 한다는 요건이 명세됐는가? [Gap, Spec §US9]
- [ ] CHK026 - 이미 다른 계정에서 인증 완료된 이메일을 재사용하려는 시도(경쟁 조건 포함)에 대한 요건이 명세됐는가? [Gap, Spec §US9]

---

## 소셜 인증 보안 요구사항

- [ ] CHK027 - 소셜 provider로부터 수신한 토큰/코드의 진위(authenticity) 검증 요건이 명세됐는가, 아니면 구현에서 암묵적으로 처리하도록 남겨졌는가? [Gap, Spec §US3]
- [ ] CHK028 - OAuth 인증 흐름의 CSRF 방지(state 파라미터 등) 요건이 명세됐는가? [Gap, Spec §US3]
- [ ] CHK029 - 소셜 provider 계정이 비활성화·삭제된 이후 해당 SocialConnection으로 로그인 시도 시 동작이 명세됐는가? [Gap, Spec §Key Entities]
- [ ] CHK030 - 소셜 가입 계정이 "즉시 인증 상태"로 처리되는 근거(provider 검증 신뢰)가 요구사항에 명시됐는가, 아니면 가정으로만 남아 있는가? [Assumption, Spec §FR-024]

## 위협 시나리오 & 감사 요구사항

- [ ] CHK031 - 보안 관련 이벤트(로그인 실패, 계정 잠금, 토큰 재사용 감지, 비밀번호 재설정)의 감사 로그 요건이 명세됐는가? (Constitution §VI: 구조적 로깅 원칙 준수) [Gap, Spec §Constitution §VI]
- [ ] CHK032 - 감사 로그에 기록할 항목과 절대 기록하지 않을 민감 정보(비밀번호, 토큰 원문)가 요구사항에 명시됐는가? [Gap, Spec §Constitution §VI]
- [ ] CHK033 - 서로 다른 기기에서 동시에 로그인 시도할 때 RefreshToken 생성 경쟁 조건(race condition)에 대한 요건이 명세됐는가? [Gap, Spec §FR-008]
- [ ] CHK034 - 계정 잠금·이메일 미인증·SUSPENDED 상태 각각에서 비밀번호 재설정 코드 발급 요청의 허용 여부가 일관되게 명세됐는가? [Consistency, Spec §FR-019, §FR-021, §Edge Cases]
- [ ] CHK035 - 토큰 전송 채널 보안(HTTPS 강제 등)에 대한 요건이 명세 또는 Assumptions에 포함됐는가? [Gap]
