# Feature Specification: 계정·인증·온보딩·인가

**Feature Branch**: `002-account-auth-onboarding`

**Created**: 2026-06-11

**Status**: Draft

## User Scenarios & Testing

### User Story 1 — 이메일 회원가입 (Priority: P1)

신규 사용자가 이메일과 비밀번호를 입력하고, 필수 약관(서비스 이용약관·개인정보 처리방침)과 만 14세 이상 연령에 동의한 후 계정을 생성한다. 마케팅 수신은 선택 동의한다. 가입 완료 시 인증 세션이 발급된다.

**Why this priority**: 가입이 없으면 개인화 서비스 전체가 성립하지 않는다. 이메일 경로는 가장 기본적인 진입점이다.

**Independent Test**: 이메일·비밀번호·약관 동의 데이터만으로 가입 API를 호출해 계정 생성 및 세션 발급을 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 미가입 이메일과 유효한 비밀번호(영문+숫자 8자 이상), 필수 약관 동의, 연령 동의 **When** 가입 요청 **Then** 계정 생성, 인증 세션 발급, 동의 이력(버전·시각) 저장
2. **Given** 이미 가입된 이메일 **When** 가입 요청 **Then** 이메일 중복 오류 반환 (409)
3. **Given** 필수 약관 미동의 **When** 가입 요청 **Then** 가입 거부 (422)
4. **Given** 만 14세 미만 연령 동의 미체크 **When** 가입 요청 **Then** 가입 거부 (422)
5. **Given** 비밀번호가 영문·숫자 미포함 또는 7자 이하 **When** 가입 요청 **Then** 가입 거부 (422)

---

### User Story 2 — 이메일 로그인 (Priority: P1)

가입된 사용자가 이메일+비밀번호로 로그인하여 인증 세션을 발급받는다.

**Why this priority**: 기존 계정 재진입의 기본 경로다.

**Independent Test**: 가입된 계정으로 로그인 API 호출 후 세션 발급 및 잘못된 자격 증명 거부를 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 유효한 이메일+비밀번호 **When** 로그인 요청 **Then** 인증 세션(액세스 토큰·리프레시 토큰) 발급
2. **Given** 틀린 비밀번호 **When** 로그인 요청 **Then** 인증 실패 (401)
3. **Given** 소셜로만 가입된 계정의 이메일 **When** 비밀번호 로그인 요청 **Then** 인증 실패 (소셜 전용 계정임을 안내)
4. **Given** 정지(SUSPENDED) 상태의 계정 **When** 로그인 요청 **Then** 접근 거부 (403)
5. **Given** 동일 계정에 연속 5회 로그인 실패 **When** 다음 로그인 시도 **Then** 계정 잠금 안내 반환, 30분 후 자동 해제

---

### User Story 3 — 소셜 가입·로그인 (Priority: P1)

사용자가 카카오·구글·애플 소셜 계정으로 가입하거나 기존 소셜 계정으로 로그인한다. 최초 소셜 인증 시 약관 동의와 연령 동의를 거쳐 계정이 생성된다.

**Why this priority**: 소셜 인증은 현대 앱의 주요 진입 경로이며 비밀번호 없는 사용자 경험을 제공한다.

**Independent Test**: 소셜 provider의 인증 결과(provider + 외부 식별자 + 이메일)를 모의 입력해 가입/로그인 분기를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 신규 소셜 계정(provider+식별자 미등록) + 필수 약관 동의 **When** 소셜 가입 요청 **Then** 계정 생성, SocialConnection 저장, 인증 세션 발급
2. **Given** 기존 등록된 소셜 계정(provider+식별자 일치) **When** 소셜 로그인 요청 **Then** 기존 계정으로 인증 세션 발급
3. **Given** 소셜 가입 시 동일 이메일이 이메일 계정으로 이미 존재 **When** 소셜 가입 요청 **Then** 오류 반환(이메일 계정 이미 존재, 연결 기능 미제공)
4. **Given** 소셜 가입 시 필수 약관 미동의 **When** 요청 **Then** 가입 거부 (422)

---

### User Story 4 — 역할 기반 접근 제어 (Priority: P1)

ADMIN 역할 사용자만 `/admin/*` 엔드포인트에 접근할 수 있다. USER 또는 미인증 요청은 거부된다. 최초 관리자 계정은 시드 또는 설정으로 부여된다.

**Why this priority**: 001에서 미보호 노출됐던 관리자 엔드포인트의 영구 보안 해소다.

**Independent Test**: 미인증·USER·ADMIN 세 가지 자격으로 `/admin/*` 호출 시 각각 401·403·200 반환 여부를 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 미인증 요청 **When** `/admin/*` 호출 **Then** 401 반환
2. **Given** USER 역할 토큰 **When** `/admin/*` 호출 **Then** 403 반환
3. **Given** ADMIN 역할 토큰 **When** `/admin/*` 호출 **Then** 200 응답
4. **Given** USER 역할 토큰 **When** 일반 보호 엔드포인트(예: `/api/v1/me`) 호출 **Then** 200 응답

---

### User Story 5 — 비밀번호 재설정 (Priority: P2)

비밀번호를 잊은 이메일 가입 사용자가 가입 이메일로 6자리 인증 코드를 받아 본인 확인 후 새 비밀번호를 설정한다.

**Why this priority**: 자기 서비스 복구 수단이 없으면 이탈률이 높아진다. 소셜 전용 사용자는 대상 외.

**Independent Test**: 코드 발급→검증→변경 3단계를 순서대로 호출해 완료 및 각 오류 케이스를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 등록된 이메일, 이메일 서비스 정상 **When** 코드 발급 요청 **Then** 6자리 코드 생성·만료 시각 설정, 이메일 발송 트리거(발송 서비스는 외부), 재전송 한도 1 차감
1a. **Given** 등록된 이메일, 이메일 서비스 장애 **When** 코드 발급 요청 **Then** 코드 미생성, 503 반환, 재전송 한도 미차감
2. **Given** 유효한 코드 + 만료 전 **When** 코드 검증 요청 **Then** 검증 성공, 변경 허가 토큰 발급
3. **Given** 만료된 코드 **When** 코드 검증 요청 **Then** 만료 오류 (410)
4. **Given** 잘못된 코드 **When** 코드 검증 요청 **Then** 오류 (401), 시도 횟수 증가
5. **Given** 재전송 제한(시간당 5회) 초과 **When** 코드 재발급 요청 **Then** 요청 차단 (429)
6. **Given** 소셜 전용 계정 이메일 **When** 코드 발급 요청 **Then** 동일 202 반환(계정 열거 방지), "비밀번호 재설정 불가 — 소셜 로그인 이용" 안내 이메일 발송(코드 미생성, FR-026)
7. **Given** 비밀번호 재설정 완료 후 이전에 발급된 리프레시 토큰 **When** 액세스 토큰 갱신 시도 **Then** 401 반환 (세션 무효화 확인)

### User Story 6 — 온보딩 (Priority: P2)

가입 직후 사용자가 프로필(닉네임·연령대·직업), 관심 카테고리(최소 3개), 팔로우 키워드(기업·테마·인물), 읽는 방식(요약 깊이·소비 방식), 브리핑 설정(시간대·음성·푸시 동의)을 입력하면 계정에 저장되고 개인화가 활성화된다.

**Why this priority**: 개인화 피드와 브리핑의 기반 데이터다. 온보딩 없이는 개인화 서비스가 작동하지 않는다.

**Independent Test**: 온보딩 데이터를 저장하는 API를 호출해 저장 및 조회를 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 인증된 사용자, 관심 카테고리 3개 이상 **When** 온보딩 데이터 제출 **Then** 전체 저장, 개인화 활성화 플래그 true
2. **Given** 인증된 사용자, 관심 카테고리 2개 이하 **When** 온보딩 제출 **Then** 오류 반환 (422, 최소 3개 필요)
3. **Given** 온보딩 미완료 상태로 앱 재진입 **When** 온보딩 상태 조회 **Then** 미완료 상태 반환
4. **Given** 브리핑 설정에서 푸시 동의 **When** 저장 **Then** 동의 여부와 시각이 기록됨

---

### User Story 7 — 프로필·관심사·설정 조회·수정 (Priority: P3)

로그인한 사용자가 자신의 프로필, 관심사, 키워드, 읽는 방식, 브리핑 설정을 조회하고 변경한다.

**Why this priority**: 가입 후 선호가 바뀌는 것은 일반적이며, 이를 지원해야 장기 사용이 가능하다.

**Independent Test**: 조회 및 수정 API를 순서대로 호출해 변경 반영 여부를 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 인증된 사용자 **When** 프로필 조회 **Then** 저장된 닉네임·연령대·직업 반환
2. **Given** 인증된 사용자 **When** 관심 카테고리 수정(최소 3개 유지) **Then** 변경 저장
3. **Given** 인증된 사용자, 관심 카테고리를 2개로 줄이려는 시도 **When** 수정 요청 **Then** 거부 (422)
4. **Given** 인증된 사용자 **When** 브리핑 시간대·음성 설정 수정 **Then** 변경 저장 및 조회 시 반영

---

### User Story 8 — 약관 버전 관리 및 재동의 (Priority: P3)

약관이 업데이트되었을 때 기존 사용자에게 재동의를 요청할 수 있다. 동의 이력은 버전별로 보관된다.

**Why this priority**: 법적 컴플라이언스 요건이다.

**Independent Test**: 새 약관 버전을 등록하고 해당 버전에 대한 동의 상태를 조회해 미동의 여부를 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 새 버전의 필수 약관이 등록됨 **When** 기존 사용자가 로그인 **Then** 재동의 필요 플래그 반환
2. **Given** 사용자가 새 버전에 동의 **When** 동의 제출 **Then** 동의 이력(버전·시각) 저장
3. **Given** 동의 이력 조회 **When** 요청 **Then** 버전별 동의 여부와 시각 목록 반환

---

### User Story 9 — 이메일 인증 (Priority: P1)

이메일로 가입한 사용자는 가입 이메일로 발송된 인증 코드를 입력해 이메일 소유를 확인해야 한다. 소셜 가입 사용자는 provider가 이미 이메일을 검증했으므로 이 단계를 거치지 않는다.

**Why this priority**: 미인증 계정이 보호된 기능에 접근하면 임시 이메일·타인 이메일 도용 위험이 있다.

**Independent Test**: 이메일 가입 후 코드 없이 보호된 API 호출 시 접근 제한 확인, 코드 검증 후 허용됨을 독립 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 이메일 가입 완료 직후 **When** 가입 요청이 처리됨 **Then** 계정 생성 트랜잭션 커밋 후 인증 코드 발송 시도, 발송 성공 시 201 + `verificationEmailSent=true`, 계정은 "미인증" 상태 유지
1a. **Given** 이메일 가입 중 이메일 서비스 장애 **When** 가입 요청이 처리됨 **Then** 계정은 생성됨(rollback 없음), 201 + `verificationEmailSent=false` 반환, 사용자는 이후 /auth/email-verification/request로 재발송 가능
2. **Given** "미인증" 계정, 유효한 인증 코드 **When** 코드 입력 **Then** 계정 "인증" 상태 전환, 보호된 기능 접근 가능
3. **Given** "미인증" 계정 **When** 보호된 기능 접근 시도 **Then** 403 + `code=EMAIL_NOT_VERIFIED` 반환 (이메일 인증 필요 안내)
4. **Given** 만료된 인증 코드 **When** 코드 입력 **Then** 만료 오류 반환, 재발송 요청 안내
5. **Given** 소셜 가입 계정 **When** 이메일 인증 상태 조회 **Then** 이미 "인증" 상태 (provider 검증으로 간주, 별도 코드 불필요)

---

### Edge Cases

- 소셜 인증 중 provider 서버 오류로 인증 실패 시 명확한 오류 안내 반환
- 애플 로그인에서 "이메일 숨기기" 선택 시 중계 이메일을 식별자로 사용
- 비밀번호 재설정 코드 검증 중 계정이 정지(SUSPENDED)되면 재설정 진행 차단
- 온보딩 미완료 상태에서도 보호된 API 접근은 허용 (온보딩은 선택적 단계)
- 리프레시 토큰 만료 후 액세스 토큰 갱신 시도 시 재로그인 요구 (401)
- 이메일 대소문자 처리: 동일 이메일의 대소문자 변형을 같은 계정으로 간주
- 계정 잠금 상태에서 올바른 비밀번호로 시도해도 잠금 해제 전까지 로그인 불가
- 무효화된 리프레시 토큰 재사용 감지 시: grace 30초 이내면 멱등 처리, 초과이면 해당 family 즉시 무효화, 5분 내 2회 이상 발생 시 계정 전체 세션으로 에스컬레이션 후 재로그인 요구 (FR-020)

---

## Requirements

### Functional Requirements

- **FR-001**: 계정은 이메일 + 비밀번호(이메일 가입) 또는 소셜 연결(소셜 가입) 중 하나로 생성된다
- **FR-002**: 가입 시 필수 약관(서비스 이용약관·개인정보 처리방침)과 만 14세 이상 연령 동의가 필수이며, 마케팅 동의는 선택이다
- **FR-003**: 동의 이력은 약관 버전, 동의 여부, 동의 시각을 포함하여 보관된다
- **FR-004**: 비밀번호는 영문·숫자를 각각 1자 이상 포함한 8자 이상이어야 한다
- **FR-005**: 이메일은 계정당 고유하며 대소문자를 구별하지 않는다
- **FR-006**: 소셜 가입 계정은 비밀번호를 보유하지 않으며 비밀번호 재설정 대상이 아니다. 해당 계정 이메일로 재설정 요청이 오면 HTTP 응답은 202(균일, FR-026 준수)이되, 내부적으로 "소셜 로그인 이용" 안내 이메일을 발송하며 재설정 코드는 생성하지 않는다
- **FR-007**: 소셜 가입 시 동일 이메일로 이미 이메일 계정이 존재하면 오류를 반환한다 (계정 연결 기능은 이 사이클 범위 밖)
- **FR-008**: 인증 성공 시 액세스 토큰과 리프레시 토큰을 발급한다. 한 계정에서 여러 기기의 동시 세션을 허용하며, 기기별로 독립된 리프레시 토큰이 발급된다
- **FR-009**: 비밀번호 재설정은 코드 발급 → 코드 검증 → 비밀번호 변경의 3단계로 동작한다
- **FR-010**: 비밀번호 재설정 코드는 6자리이며 15분 후 만료된다
- **FR-011**: 비밀번호 재설정 코드 재전송은 시간당 5회로 제한된다. 한도는 이메일 발송 성공 시에만 차감되며, 외부 이메일 서비스 오류로 발송이 실패하면 코드가 생성되지 않고 503을 반환하며 한도에 산입하지 않는다
- **FR-012**: 잘못된 코드 입력 횟수가 일정 한도를 초과하면 해당 코드를 무효화한다
- **FR-013**: 온보딩 데이터(프로필·관심 카테고리·팔로우 키워드·읽는 방식·브리핑 설정)를 계정에 저장·조회·수정할 수 있다
- **FR-014**: 관심 카테고리는 최소 3개를 선택해야 개인화 활성화 플래그가 true가 된다
- **FR-015**: 계정은 USER 또는 ADMIN 역할을 가지며, 기본값은 USER다
- **FR-016**: ADMIN 역할만 `/admin/*` 엔드포인트에 접근할 수 있다. 미인증 요청은 401, USER 요청은 403을 반환한다
- **FR-017**: 최초 관리자 계정은 DB 시드 또는 서버 설정으로 부여되며 사용자 대상 역할 변경 UI는 이 사이클 범위 밖이다
- **FR-018**: 약관은 버전·유형·시행일을 가지며, 새 버전 등록 시 기존 사용자에게 재동의 필요 여부를 표시할 수 있다
- **FR-019**: 동일 계정에 연속 5회 로그인 실패 시 계정을 일시 잠금하고 잠금 안내를 반환한다. 잠금은 30분 후 자동 해제된다
- **FR-020**: 리프레시 토큰은 사용 시 즉시 무효화되고 새 토큰이 발급된다(Refresh Token Rotation). 이미 무효화된 리프레시 토큰의 재사용이 감지된 경우, (a) 소비 후 30초 이내 재사용은 네트워크 재시도로 간주해 멱등 처리(무효화하지 않음), (b) 그 외에는 해당 토큰의 **family(기기 세션 체인)를 즉시 무효화**하고, (c) 동일 계정에서 5분 내 family 무효화가 2회 이상이면 **계정 전체 세션으로 에스컬레이션**한다
- **FR-021**: 이메일 가입 시 인증 코드를 가입 이메일로 발송하며, 계정은 코드 검증 전까지 "미인증" 상태를 유지한다
- **FR-022**: "미인증" 상태의 계정은 보호된 기능(피드 조회, 온보딩 저장 등)에 대한 접근이 제한된다
- **FR-023**: 이메일 인증 코드는 비밀번호 재설정 코드와 동일한 정책을 따른다 — 만료 15분, 재전송 시간당 5회(발송 성공 시에만 차감), 오입력 5회 초과 시 코드 무효화
- **FR-024**: 소셜 가입 계정은 이메일 인증 단계를 거치지 않으며 가입 시점에 이미 "인증" 상태로 생성된다
- **FR-025**: 비밀번호 변경 또는 비밀번호 재설정 성공 시, 해당 계정의 모든 활성 세션(리프레시 토큰)을 즉시 무효화한다
- **FR-026**: 복구 흐름(비밀번호 재설정 코드 요청, 이메일 인증 코드 요청)과 비밀번호 로그인 실패 응답은 계정·이메일 존재 여부를 누설하지 않는다. 복구 흐름은 존재·미존재 동일 응답을 반환하며, 로그인 실패는 일반화된 "인증 실패"로 처리한다. 단, 다음 두 경우는 UX상 수용하는 예외다: (1) 회원가입 시 이메일 중복 안내(409), (2) 소셜 전용 계정에 비밀번호 로그인 시도 시 소셜 로그인 안내(User Story 2 시나리오 3)
- **FR-027**: 소셜 로그인 흐름은 콜백 위조(CSRF)를 방지한다. 요청과 콜백 간 검증 가능한 일회성 값을 사용하며, 검증에 실패한 콜백은 거부한다
- **FR-028**: 보안 관련 이벤트(로그인 실패, 계정 잠금, 비밀번호·역할 변경, 관리자 접근, 리프레시 토큰 재사용 감지)를 감사 목적으로 기록한다. 기록 방식(구조화 로그 vs 감사 테이블)은 설계 시 결정한다

### Key Entities

- **Account**: 이메일(고유, nullable for social-only), 비밀번호 해시(nullable), 역할(USER/ADMIN), 상태(ACTIVE/SUSPENDED/DELETED — DELETED는 미래 탈퇴 기능을 위해 예약, 008 구현), 온보딩 완료 여부, 가입 경로
- **SocialConnection**: Account 참조, provider(KAKAO/GOOGLE/APPLE), 외부 식별자(provider 내 고유), 연결 시각
- **TermsVersion**: 약관 유형(SERVICE/PRIVACY/MARKETING), 버전 식별자, 시행일, 필수 여부
- **ConsentRecord**: Account 참조, TermsVersion 참조, 동의 여부, 동의 시각
- **RefreshToken**: Account 참조, 토큰 해시, 기기 식별자(선택), 발급 시각, 만료 시각, 무효화 여부, 무효화 시각 — 계정당 N개 존재 가능(기기별 독립); Rotation으로 사용 즉시 무효화
- **VerificationCode**: Account 참조, purpose(EMAIL_VERIFY / PASSWORD_RESET), 코드 해시(SHA-256), 만료 시각, 입력 시도 횟수, 발급 횟수(시간 단위), 윈도우 시작 시각, 사용 여부. 단일 엔티티로 통합(plan.md 결정). PasswordResetCode·EmailVerificationCode는 이 엔티티의 purpose 값으로 구분됨
- **UserProfile**: Account 참조, 닉네임, 연령대, 직업 (1:1)
- **UserInterests**: Account 참조, 카테고리 목록
- **FollowKeyword**: Account 참조, 키워드 텍스트, 유형(COMPANY/THEME/PERSON)
- **ReadingPreference**: Account 참조, 요약 깊이(BRIEF/BALANCED/DEEP), 소비 방식(READ/LISTEN/BOTH)
- **BriefingSettings**: Account 참조, 브리핑 시간대(HH:MM), 음성 활성화 여부, 푸시 동의 여부, 푸시 동의 시각

## Success Criteria

### Measurable Outcomes

- **SC-001**: 이메일·소셜 어느 경로로도 가입·로그인이 60초 내 완료되고 인증 세션이 발급된다
- **SC-002**: 필수 약관 미동의, 만 14세 미만 미확인, 약한 비밀번호(정책 미충족) 세 경우 모두 가입이 거부된다
- **SC-003**: 비밀번호 재설정이 코드 발급→검증→변경 순서로 완료되며, 코드 만료·오입력·재전송 초과가 각각 올바른 오류로 처리된다
- **SC-004**: 온보딩에서 입력한 모든 항목(프로필·관심사·키워드·읽는 방식·브리핑 설정)이 저장되고 이후 조회 시 동일하게 반환된다
- **SC-005**: 관심 카테고리 3개 이상 선택 시 개인화 활성화 플래그가 true로 전환된다
- **SC-006**: `/admin/*` 엔드포인트가 ADMIN 역할에만 200을 반환하고, USER는 403, 미인증은 401을 반환한다
- **SC-007**: 동일 이메일로 계정 중복 생성이 거부된다
- **SC-008**: 약관 동의 이력이 버전·시각 단위로 조회 가능하다
- **SC-009**: 이메일 가입 계정은 코드 검증을 통과해야 "인증" 상태가 되며, 미인증 상태에서는 보호된 기능 접근이 제한된다. 소셜 가입 계정은 가입 즉시 인증 상태로 설정된다
- **SC-010**: 비밀번호 변경 또는 비밀번호 재설정 성공 후, 해당 계정에서 이전에 발급된 모든 리프레시 토큰이 무효화되어 갱신 시도 시 401이 반환된다

## Assumptions

- 이메일 발송(코드 전달)은 외부 이메일 서비스에 위임한다. 이 스펙은 발송 트리거만 포함하며 실제 발송 서비스는 범위 밖이다.
- 비밀번호 재설정 코드 만료 기간은 15분, 시간당 재전송 상한은 5회를 합리적 기본값으로 채택한다.
- 소셜 인증에서 provider가 이메일을 반환하지 않을 경우(예: 애플 이메일 숨기기의 중계 이메일 포함) 내부 식별자로 계정을 생성하고 이메일은 nullable로 처리한다.
- 소셜 계정과 이메일 계정 연결(Account Linking) 기능은 이 사이클 범위 밖이다. 동일 이메일 충돌 시 오류를 반환한다.
- 온보딩은 가입 후 진행하는 별도 단계이며, 온보딩 미완료 상태에서도 인증된 API 접근은 허용된다.
- 액세스 토큰 유효 기간은 1시간, 리프레시 토큰은 30일을 합리적 기본값으로 채택한다.
- ADMIN 역할 부여는 시드 SQL 또는 서버 설정(환경변수)으로만 수행한다. 사용자 대상 역할 변경 워크플로는 spec 008 범위다.
- 회원 탈퇴 기능(DELETED 상태 전환, 개인정보 파기 정책)은 spec 008로 deferred. 이 사이클은 DELETED 상태 값만 스키마에 예약한다.
- 모든 시각은 UTC로 저장하며, 브리핑 시간대는 사용자 기기의 타임존 오프셋을 함께 저장한다.
- 잘못된 코드 입력 상한은 5회로 설정하며, 초과 시 해당 코드를 즉시 무효화한다.
