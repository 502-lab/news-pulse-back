## Clarifications

### Session 2026-06-08

- Q: 중복 기사 판단 기준 (소스+URL vs URL only) → A: 원문 URL을 단일 식별 키로 사용. 중복 감지 시 폐기가 아닌 병합 처리 — 어떤 출처들이 해당 기사를 수집했는지(provenance)와 최초 수집 시각을 보존하며, 중복 감지 건수는 집계 가능하게 기록한다.
- Q: AI 요약·분류 처리 타이밍 (수집 동기 vs 비동기 분리) → A: 수집 스케줄러는 기사 저장(PENDING 상태)만 담당. 별도 AI 처리 스케줄러가 PENDING 큐를 폴링해 분류·요약을 처리하고 COMPLETED/FAILED로 전이. FAILED는 설정된 횟수만큼 재시도 후 영구 실패로 표시. 분류와 요약의 처리 상태는 각각 독립적으로 추적한다.
- Q: 기사 데이터 보존 기간 → A: 기본 90일, 환경변수로 조정 가능. 만료 기사는 피드 노출 제외 후 물리 삭제(2단계). 사용자가 저장한 기사는 자동 삭제 제외(최소 요약·메타 보존). 트렌드 집계 결과는 원본 기사와 독립 저장하여 기사 정리가 과거 통계에 영향을 주지 않도록 한다.
- Q: 피드 페이지네이션 방식 및 페이지 크기 → A: 커서 기반 페이지네이션 채택(오프셋 방식의 페이지 드리프트 방지). 기본 페이지 크기 20건, 최대 100건. 최대값 초과 요청은 거부하지 않고 100건으로 clamp. 기본값·최대값은 API 계약에 명시해 클라이언트가 신뢰 가능하게 한다. 값은 설정으로 조정 가능하되 기본값·최대값은 계약에 고정 문서화한다.
- Q: 수집 주기 기본값 → A: 전역 기본값 15분(설정으로 조정 가능). 출처별 개별 주기 덮어쓰기 허용. 각 출처의 rate limit을 준수하며, 한도가 빠듯한 출처는 별도 호출 예산(call budget)으로 관리. SC-001의 "수집 주기 내 반영" SLO는 설정된 주기를 기준으로 정의하며 숫자를 하드코딩하지 않는다.
- Q: 피드 API 응답 시간 목표 → A: 서버 측 P95 응답 시간 1초 이내. 측정 범위는 서버 수신~응답 완료(클라이언트 네트워크·렌더링 제외), 예상 동시 사용자 부하 조건 하 측정. 목표 달성 방식(캐시 등)은 명세에서 규정하지 않으며 plan 단계에서 결정한다.
- Q: 서비스 처리 볼륨 규모 → A: 하루 수집 기사 10,000건을 설계 헤드룸으로 채택(실제 24개 RSS 출처 수집량은 그 이하). SC-009의 SLO 부하 조건은 단일 인스턴스 기준 동시 사용자 100-200명으로 정의. 동시 사용자 1,000명 이상 대응은 캐시·수평 확장으로 처리할 성장 목표이며 이번 사이클 범위 밖.
- Q: 기사 상세 응답의 요약 깊이 반환 방식 → A: 기사 상세 응답은 항상 brief/balanced/deep 세 슬롯을 각 상태(COMPLETED+본문 / PENDING / 미생성)와 함께 반환한다. balanced는 AI 처리 스케줄러가 수집 직후 생성하고(eager), brief/deep는 최초 상세 조회 시 생성·캐시한다(lazy). article-level summary_status=COMPLETED는 balanced 생성 완료를 의미한다. US3 본문의 "선택한 깊이"는 별도 depth 파라미터가 아니라 반환된 슬롯 중 클라이언트가 선택해 표시하는 것이다.
- Q: 피드 목록 노출 최소 조건 → A: category_status ∈ {COMPLETED, FAILED}인 기사만 피드 노출, PENDING 제외. category_status=FAILED 기사는 API 응답에서 카테고리를 '기타(OTHER)'로 반환해 피드에서 영구히 숨기지 않는다. summary_status는 노출 조건과 무관하며(COMPLETED/FAILED 모두 노출), 요약 FAILED 기사는 "요약 없음" 상태로 표기된다. SC-001 SLO 시간에는 분류 처리 시간이 포함된다.
- Q: 피드 목록 요약 미리보기 방식 → A: 피드 목록(카드)에는 항상 eager 생성되는 balanced 요약을 사용한다. 클라이언트가 1~2줄로 트런케이트해 표시하며 별도 피드용 길이 조정은 하지 않는다. brief는 상세 화면 전용 슬롯으로 lazy 유지(Q8과 일관). 사용자의 요약 깊이 선호는 상세 조회에서 적용한다. AS1에서 "brief 요약" → "balanced 요약(목록용 미리보기)"으로 반영.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 자동 뉴스 수집 및 중복 제거 (Priority: P1)

시스템은 등록된 뉴스 출처들을 주기적으로 폴링하여 새 기사를 수집한다.
이미 수집된 기사와 원문 URL이 같은 기사는 새 레코드를 만들지 않고 병합 처리하여(새 출처를 provenance에 추가), 처음 보는 기사만 신규 저장한다.
출처 중 일부가 응답하지 않거나 오류를 반환해도 나머지 출처의 수집은 계속 진행한다.

**Why this priority**: 수집 파이프라인이 없으면 피드에 표시할 기사가 없다.
전체 서비스의 데이터 공급 기반이 되며, 중복 방지는 피드 품질의 핵심이다.

**Independent Test**: 수집 스케줄러를 한 번 실행한 뒤, 특정 기사가 정확히 한 번만
저장되었음을 확인하고, 특정 출처를 오프라인으로 두었을 때 나머지 출처의 기사가 정상
저장됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 등록된 뉴스 출처에 새 기사 N건이 게시되어 있을 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 해당 N건이 시스템에 저장되고, 각 기사에 수집 시각이 기록된다.
2. **Given** 이미 수집된 기사와 동일한 URL의 기사가 다른 출처에서 수집될 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 기존 기사 레코드는 그대로 유지되고, 새 출처가 해당 기사의 ArticleSource(provenance)에 추가되며 병합 건수가 통계에 집계된다.
3. **Given** 출처 A는 정상, 출처 B는 타임아웃 상태일 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 출처 A의 기사는 정상 수집되고, 출처 B의 실패가 로그에 기록되며 전체 수집 작업은 완료 상태로 종료된다.

### User Story 2 - 카테고리 분류 및 AI 요약 생성 (Priority: P2)

수집 스케줄러가 저장한 PENDING 상태 기사를 별도 AI 처리 스케줄러가 폴링하여,
카테고리 분류와 AI 요약(brief/balanced/deep)을 순차 처리한다.
분류와 요약은 각각 독립적인 처리 상태(PENDING/COMPLETED/FAILED)를 가지며,
한쪽 실패가 다른 쪽 처리를 막지 않는다.
FAILED 상태는 설정된 최대 재시도 횟수까지 재처리되며, 한도 초과 시 영구 실패로 표시된다.
처리 완료된 요약에는 생성 시각과 AI 생성 식별 메타정보가 포함된다.
AI 서비스 장애·지연은 수집 파이프라인에 영향을 주지 않으며, 상태는 DB 기반으로 관리되어
재시작·중복 실행에도 멱등하게 동작한다.

**Why this priority**: 카테고리와 요약은 앱 사용자가 기사를 빠르게 판단하는 핵심 가치다.
수집(P1) 이후 즉시 처리되어야 피드에서 의미 있는 정보를 제공할 수 있다.

**Independent Test**: PENDING 상태 기사 1건에 대해 AI 처리 스케줄러를 실행한 뒤,
category_status=COMPLETED·summary_status=COMPLETED가 되고 balanced 요약이 저장됨을 확인한다(brief/deep는 미생성 상태로 반환되며 최초 상세 조회 시 생성됨).
요약 서비스가 오류를 반환할 때 summary_status=FAILED(category_status는 독립적으로 유지)임을 확인한다.
최대 재시도 횟수 초과 시 영구 실패 상태가 됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 수집 스케줄러가 기사를 저장해 category_status=PENDING, summary_status=PENDING인 기사가 있을 때,
   **When** AI 처리 스케줄러가 실행되면,
   **Then** category_status=COMPLETED(카테고리 1개 부여)·summary_status=COMPLETED(balanced 요약 저장·생성 시각 기록, brief/deep는 미생성 상태)로 전이된다.
2. **Given** AI 요약 서비스가 오류를 반환하지만 분류 서비스는 정상일 때,
   **When** AI 처리 스케줄러가 실행되면,
   **Then** category_status=COMPLETED(카테고리 부여)·summary_status=FAILED로 각각 기록되고, 기사는 카테고리와 함께 피드에 노출된다.
3. **Given** summary_status=FAILED 기사가 최대 재시도 횟수를 초과했을 때,
   **When** AI 처리 스케줄러가 재실행되면,
   **Then** 해당 기사는 재시도 대상에서 제외되고 summary_status=FAILED(영구) 상태로 유지된다.
4. **Given** 카테고리 분류 결과가 없거나 모호할 때,
   **When** 분류 처리가 실행되면,
   **Then** 기사에 "기타(OTHER)" 카테고리가 부여되고 category_status=COMPLETED로 전이된다.

### User Story 3 - 앱 사용자의 카테고리 피드 조회 (Priority: P3)

앱 사용자(인증된 사용자)는 최신순 뉴스 목록 또는 특정 카테고리로 필터링한 뉴스 목록을 조회한다.
각 기사 항목에는 제목, 카테고리, 출처, 발행 시각, balanced 요약(목록용 미리보기)이 포함된다.
기사 상세 조회 시 원문 링크, 기자, 수집 시각, 세 가지 깊이의 요약 슬롯(각 상태 포함), AI 생성 면책 안내,
요약 생성 시각이 제공된다. 클라이언트는 반환된 슬롯 중 원하는 깊이를 선택해 표시하며, 별도 depth 파라미터는 없다.
커서 기반 페이지네이션(기본 20건, 최대 100건 clamp)을 통해 대량의 기사를 안정적으로 조회한다. 수집 중 새 기사가 추가되어도 페이지 드리프트 없이 일관된 결과를 반환한다.

**Why this priority**: 피드 조회는 앱 사용자가 직접 소비하는 핵심 기능으로,
수집(P1)·분류/요약(P2) 이후에 의미 있는 데이터를 제공할 수 있다.

**Independent Test**: API를 통해 카테고리 필터 없이 피드를 조회했을 때 최신순으로
기사 목록이 반환되는지, 특정 카테고리로 필터링 시 해당 카테고리 기사만 반환되는지 확인한다.
기사 상세 API 호출 시 원문 링크와 요약이 포함되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 여러 카테고리의 기사가 저장되어 있을 때,
   **When** 인증된 사용자가 카테고리 필터 없이 피드를 조회하면,
   **Then** 기사가 원 출처 발행 시각 기준 최신순으로 반환되고, 각 항목에 제목·카테고리·출처·발행 시각·balanced 요약(목록용 미리보기)이 포함된다.
2. **Given** 경제·금융 카테고리 기사 5건이 저장되어 있을 때,
   **When** 인증된 사용자가 "경제·금융" 카테고리로 피드를 조회하면,
   **Then** 해당 카테고리 기사 5건만 반환된다.
3. **Given** 기사 ID가 유효할 때,
   **When** 인증된 사용자가 기사 상세를 조회하면,
   **Then** 응답에 원문 링크·기자·수집 시각·세 가지 깊이의 요약(상태 포함)·AI 생성 면책 안내·요약 생성 시각이 포함된다.
4. **Given** 요약 생성이 실패한 기사를 조회할 때,
   **When** 사용자가 해당 기사 상세를 조회하면,
   **Then** 요약 상태가 FAILED로 반환되고, 원문 링크가 함께 제공된다.
5. **Given** 피드에 기사가 50건 있고 페이지 크기를 150으로 요청할 때,
   **When** 인증된 사용자가 피드를 조회하면,
   **Then** 최대값 100건으로 clamp되어 처리되고, 실제 50건과 다음 페이지 없음(cursor=null)이 반환된다.

### User Story 4 - 관리자 파이프라인 모니터링 (Priority: P4)

관리자(인증된 관리자 역할)는 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율,
중복 제거율, 파이프라인 단계별 처리 현황을 API로 조회한다.
이 데이터는 백오피스 UI가 소비하는 API로 제공된다.

**Why this priority**: 운영 모니터링은 서비스 품질 유지에 필수이나,
사용자 기능(P1-P3) 완성 후 제공해도 서비스 운영에 지장이 없다.

**Independent Test**: 관리자 권한 토큰으로 대시보드 API를 호출하여
오늘 수집 건수·요약 완료율·병합 처리 건수가 실제 저장된 데이터와 일치하는지 확인한다.
일반 사용자 권한으로 호출 시 접근 거부됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 오늘 기사 100건이 수집되고 그 중 80건 요약 완료, 5건 중복 제거된 상태일 때,
   **When** 관리자가 대시보드 API를 조회하면,
   **Then** 수집 100건, 요약 완료율 80%, 병합 처리 5건, 카테고리별 기사 수가 정확히 반환된다.
2. **Given** 일반 사용자 토큰으로 관리자 API를 호출할 때,
   **When** 대시보드 엔드포인트에 요청을 보내면,
   **Then** 접근 거부(403) 응답이 반환된다. *(인증 구현 spec 002 이후 달성 가능, 001 범위 밖)*

### Edge Cases

- 뉴스 출처가 0건 등록된 상태에서 수집 스케줄러가 실행될 때: 수집 작업이 정상 완료되고 처리 건수 0으로 기록된다.
- 특정 출처가 연속으로 실패할 때: 해당 출처의 오류 횟수가 누적 기록되며, 다른 출처 수집에 영향을 주지 않는다.
- AI 서비스 전체 장애 시: 수집 스케줄러는 계속 기사를 저장하고(category_status=PENDING, summary_status=PENDING), AI 처리 스케줄러는 FAILED 처리 후 재시도 카운트를 누적한다. AI 서비스 복구 후 retry_limit 미초과 기사는 자동으로 재처리된다.
- 기사 원문이 삭제되거나 접근 불가능해진 경우: 시스템은 수집 시점에 저장된 메타데이터(제목·발행 시각·출처)와 원문 링크를 유지하고, 원문 재수집을 시도하지 않는다.
- 동일 기사가 두 출처에서 동시에 수집될 때: 원문 URL 기준으로 중복 감지 후 병합 처리한다. 기사 레코드는 1건으로 유지되며, 두 번째 출처는 해당 기사의 provenance에 추가된다. 병합 건수는 파이프라인 통계에 집계된다.
- 만료(feed_visible=false)된 기사와 동일 정규화 URL이 재수집될 때: dedup이 기존 레코드에 매칭되면 provenance(ArticleSource)와 last-seen 정보만 갱신하고, feed_visible은 변경하지 않는다(재활성화 없음). 이미 물리 삭제된 기사와 동일 URL이 재수집될 경우 신규 기사로 삽입된다.
- 피드 조회 시 조회 결과가 0건인 경우: 빈 목록과 함께 정상 응답(200)이 반환된다.
- 보존 기간이 지난 기사를 사용자가 저장한 경우: 피드 노출에서는 제외되지만 사용자 저장 목록에서는 접근 가능하며 요약·메타정보가 보존된다.
- 만료 예정 기사에 AI 처리가 아직 진행 중인 경우: 피드 노출 제외 처리 후 AI 처리가 완료되거나 만료되면 정리 대상에 포함된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 설정된 전역 수집 주기(기본값 15분)에 따라 등록된 모든 활성 뉴스 출처에서 기사를 수집해야 한다. 출처별 개별 수집 주기를 설정해 전역 기본값을 덮어쓸 수 있어야 한다. 각 출처의 rate limit을 준수해야 하며, 호출 한도가 빠듯한 출처는 별도 일일 호출 예산(call_budget_daily)으로 관리해야 한다. 일일 예산 초과 시 해당 공급자 호출만 중단하고 대상 기사는 PENDING 상태를 유지하여 다음 예산 창에서 재개한다. 예산 소진 이벤트는 로그·메트릭으로 기록하며, AI 처리 실패(FAILED 전이)와 구분된다. 모든 주기 설정값은 코드에 하드코딩하지 않는다.
- **FR-002**: 시스템은 원문 URL을 단일 식별 키로 사용해 중복 수집을 방지해야 한다. 동일 URL의 기사가 다른 출처에서 수집될 경우 폐기하지 않고 병합 처리하며, 해당 기사를 수집한 출처 목록(provenance)과 각 출처별 수집 시각을 보존해야 한다. 기사의 최초 수집 시각(first_collected_at)은 이후 병합 시 변경하지 않는다. 병합 발생 건수는 파이프라인 통계에서 집계 가능해야 한다.
- **FR-003**: 시스템은 특정 출처 수집이 실패해도 나머지 출처 수집을 계속 진행해야 한다.
- **FR-004**: 시스템은 각 기사에 원 출처의 발행 시각(published_at)과 시스템의 수집 시각(collected_at)을 구분하여 저장해야 한다.
- **FR-005**: 수집 스케줄러는 기사를 저장할 때 category_status=PENDING, summary_status=PENDING으로 초기화해야 한다. AI 처리는 별도 스케줄러가 담당하며, 수집 스케줄러는 AI 처리를 직접 호출하지 않는다.
- **FR-006**: AI 처리 스케줄러는 category_status=PENDING 기사를 폴링하여 경제·금융, 과학, 정치, 스포츠, 세계, 연예·문화, 건강·의학, 자동차, IT, 기타 중 하나로 분류하고 category_status=COMPLETED로 전이해야 한다. 분류 불가 시 기타(OTHER)로 처리한다.
- **FR-007**: AI 처리 스케줄러는 summary_status=PENDING 기사를 폴링하여 balanced 깊이의 요약을 생성하고, 생성 시각과 AI 생성 식별 메타정보를 저장한 뒤 summary_status=COMPLETED로 전이해야 한다. brief/deep 요약은 기사 상세가 최초로 요청될 때 생성 후 DB에 저장한다(lazy 생성 후 재요청 시 재사용, Redis 미도입). 각 깊이 슬롯은 독립적인 상태(COMPLETED/PENDING/FAILED/미생성)를 가지며, article-level summary_status=COMPLETED는 balanced 생성 완료를 의미한다.
- **FR-008**: 분류 또는 요약 처리가 실패하면 해당 상태를 FAILED로 기록하고, 환경변수로 설정된 최대 재시도 횟수(retry_limit)까지 재시도해야 한다. 한도 초과 시 영구 FAILED로 표시하고 더 이상 재시도하지 않는다. 분류와 요약의 실패는 독립적으로 처리된다.
- **FR-009**: AI 처리 스케줄러는 DB 상태 기반으로 동작하여, 재시작·중복 실행 시에도 동일 기사를 중복 처리하지 않아야 한다(멱등성).
- **FR-010**: 시스템은 원문 전체를 저장하거나 재배포하지 않고, 제목·요약·원문 링크·메타정보만 제공해야 한다.
- **FR-011**: 인증된 사용자는 기사 목록을 발행 시각 기준 최신순으로 조회할 수 있어야 하며, 카테고리로 필터링할 수 있어야 한다. 피드 노출 조건은 category_status ∈ {COMPLETED, FAILED}이며, PENDING 기사는 피드에서 제외된다. category_status=FAILED 기사는 카테고리를 '기타(OTHER)'로 표시한다. summary_status는 노출 조건과 무관하다(COMPLETED/FAILED 모두 노출, 요약 FAILED는 "요약 없음"으로 표기).
- **FR-012**: 인증된 사용자는 기사 상세(category_status·summary_status, 요약, 원문 링크, provenance 출처 목록, 기자, 발행/수집 시각, AI 면책 안내)를 조회할 수 있어야 한다.
- **FR-013**: 기사 목록 조회는 커서 기반 페이지네이션을 사용해야 한다. 기본 페이지 크기는 20건, 최대 100건이며, 최대값을 초과한 요청은 거부하지 않고 100건으로 clamp한다. 기본값과 최대값은 API 계약에 명시되어야 하며, 값은 설정으로 조정 가능하다. 커서는 발행 시각 기반으로 생성되어 새 기사 수집 중에도 페이지 드리프트 없이 안정적으로 동작해야 한다.
- **FR-014**: 인증된 관리자는 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율, 병합(중복 감지) 건수, 파이프라인 단계별 처리 현황을 조회할 수 있어야 한다.
- **FR-015**: 관리자 모니터링 API는 관리자 역할을 가진 사용자만 접근 가능해야 한다.
- **FR-016**: 수집 스케줄러와 AI 처리 스케줄러 각각의 시작/종료/실패 이벤트가 관측 가능하게 기록되어야 한다.
- **FR-017**: 기사 보존 기간은 환경변수로 설정 가능해야 하며 기본값은 90일이다. 보존 기간 값은 코드에 하드코딩하지 않는다.
- **FR-018**: 보존 기간이 지난 기사는 즉시 삭제하지 않고, 먼저 피드 노출 대상에서 제외한 뒤 별도 정리 작업이 물리적으로 삭제한다(2단계 만료).
- **FR-019**: 사용자가 저장(북마크)한 기사는 보존 기간 초과 시 자동 삭제 대상에서 제외되어야 하며, 삭제되더라도 제목·요약·원문 링크·메타정보는 보존되어야 한다. Article 데이터 모델은 "사용자 저장 여부"를 추적할 수 있는 구조를 갖추어야 한다. (사용자 저장·해제 액션 자체는 별도 명세에서 구현한다.)
- **FR-020**: 트렌드 집계 결과는 원본 기사 데이터와 독립적으로 저장되어야 하며, 기사 정리(만료·삭제) 작업이 과거 트렌드 통계에 영향을 주어서는 안 된다. (트렌드 집계 기능은 별도 명세에서 구현한다.)

### Key Entities *(include if feature involves data)*

- **Source (뉴스 출처)**: 기사를 수집할 외부 뉴스 제공처. 이름, 피드 URL, 활성 여부, 마지막 수집 시각, 연속 실패 횟수, 개별 수집 주기(미설정 시 전역 기본값 적용), 일일 호출 예산(call_budget_daily, rate limit 관리용)을 포함한다. 실제 사용량은 SourceDailyUsage(출처·날짜 키)로 별도 추적한다.
- **Article (기사)**: 수집된 개별 뉴스 기사. 원문 URL이 전역 식별 키(dedup 기준)이다. 제목, 원문 링크, 기자, 발행 시각(published_at), 최초 수집 시각(first_collected_at), 카테고리, category_status(PENDING/COMPLETED/FAILED), summary_status(PENDING/COMPLETED/FAILED), AI 처리 재시도 횟수(category_retry_count·summary_retry_count), 만료 예정 시각(expires_at), 피드 노출 여부(feed_visible), 사용자 저장 여부(user_saved)를 포함한다. 하나의 기사는 여러 출처에서 수집될 수 있으며, 출처 목록은 ArticleSource를 통해 관리된다.
- **ArticleSource (수집 출처 이력 / Provenance)**: Article과 Source를 연결하는 관계 엔티티. 어떤 출처에서 언제(per-source collected_at) 해당 기사를 수집했는지를 기록한다. 동일 기사가 N개 출처에서 수집되면 N개의 ArticleSource 레코드가 생성된다.
- **Summary (요약)**: 기사에 귀속되는 AI 생성 요약. 깊이(BRIEF/BALANCED/DEEP), 슬롯 상태(COMPLETED/PENDING/FAILED/미생성), 요약 본문, 생성 시각, AI 생성 식별 플래그를 포함한다. 기사당 최대 3개(깊이별 1개). balanced는 AI 처리 파이프라인에서 즉시 생성되며, brief/deep는 최초 상세 조회 요청 시 생성(lazy)된다.
- **Category (카테고리)**: 기사의 주제 분류값. 고정된 열거형(경제·금융, 과학, 정치, 스포츠, 세계, 연예·문화, 건강·의학, 자동차, IT, 기타).
- **PipelineStats (파이프라인 통계)**: 관리자 대시보드를 위한 집계 뷰. 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율, 병합 처리(중복 감지) 건수.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 출처에 새 기사가 게시된 후 해당 출처에 설정된 수집 주기(전역 기본값 15분) 내에 카테고리 피드에서 해당 기사가 조회된다. SLO 시간에는 수집부터 카테고리 분류 완료(category_status ∈ {COMPLETED, FAILED})까지의 시간이 포함된다. SLO 기준은 설정값을 따르며 명세에 숫자를 고정하지 않는다.
- **SC-002**: 동일 기사가 피드에 중복으로 노출되지 않는다 (중복 노출 건수 = 0).
- **SC-003**: 특정 출처에 장애가 발생한 상태에서도 피드 조회가 정상 동작하고 사용자에게 기사 목록이 반환된다.
- **SC-004**: 수집된 모든 기사에 카테고리가 부여된다 (카테고리 미할당 기사 비율 = 0%).
- **SC-005**: 요약 생성이 실패한 기사는 요약 상태(FAILED)가 명시된 채 피드와 상세 조회에서 확인 가능하다.
- **SC-006**: 관리자는 오늘 수집 건수·요약 완료율·병합 처리 건수·카테고리별 기사 수를 단일 API 호출로 확인할 수 있다.
- **SC-007**: 수집·분류·요약 파이프라인의 각 단계별 처리 결과가 로그에서 추적 가능하다.
- **SC-008**: 보존 기간이 지난 기사는 피드 조회에 노출되지 않는다. 사용자가 저장한 기사는 보존 기간 초과 후에도 해당 사용자에게 요약·메타정보가 제공된다.
- **SC-009**: 피드 기사 목록 조회 API의 서버 응답 시간은 단일 인스턴스 기준 동시 사용자 100-200명 부하 조건 하에서 P95 기준 1초 이내여야 한다. 측정 범위는 서버 요청 수신~응답 완료로 한정하며, 클라이언트 네트워크 지연·렌더링 시간을 포함하지 않는다. 목표 달성 방식은 구현 계획 단계에서 결정한다. *(운영 SLO — 빌드 게이트 아님, 배포 후 부하 테스트로 측정)*
- **SC-010**: 수집·분류·요약 파이프라인은 하루 최대 10,000건의 기사를 처리할 수 있어야 한다(설계 헤드룸 기준). *(설계 헤드룸 — 빌드 게이트 아님, 배포 후 측정)*

## Assumptions

- 뉴스 출처 등록·관리(Source CRUD)는 이번 사이클 범위 밖이며, 출처는 초기 데이터 또는 별도 관리 기능으로 사전 등록되어 있다고 가정한다.
- 사용자 인증 및 역할(일반 사용자 / 관리자) 관리는 별도 명세에서 다루며, 이 기능은 인증 시스템이 이미 동작 중임을 전제한다.
- 카테고리 분류와 AI 요약은 외부 AI 서비스를 통해 처리된다. AI 서비스 선택·프롬프트 설계는 구현 단계에서 결정한다.
- 전역 수집 주기 기본값은 15분이며 환경변수로 조정 가능하다. 각 출처는 개별 수집 주기와 호출 예산을 설정해 전역 기본값을 덮어쓸 수 있다.
- 기사 피드 조회는 인증된 사용자만 접근 가능하다. 공개(비인증) 피드는 이번 사이클 범위 밖이다.
- AI 요약 재처리는 AI 처리 스케줄러가 PENDING/FAILED(retry_limit 미초과) 상태를 폴링할 때 자동으로 이루어진다.
- 사용자 기사 저장(북마크) 기능은 별도 명세에서 구현된다. 이 명세에서는 Article이 "사용자 저장 여부(user_saved)"를 추적할 수 있는 구조를 갖추되, 저장·해제 액션 자체는 범위 밖이다.
- 트렌드 집계 저장소는 별도 명세에서 설계된다. 이 명세에서는 원본 기사 정리가 트렌드 통계에 영향을 주지 않아야 한다는 격리 원칙만 정의한다.
- 기사 발행 시각(published_at)은 외부 소스가 제공하는 값을 사용하며, 누락된 경우 수집 시각으로 대체한다.
- 설계 볼륨 헤드룸은 하루 수집 기사 10,000건, 단일 인스턴스 기준 동시 사용자 100-200명이다. 실제 24개 RSS 출처의 수집량은 10,000건 이내로 예상한다. 이 기준이 인덱스·배치 크기·성능 테스트 시나리오의 근거가 된다.

## Out of Scope

- 클라이언트(React/Android) UI 레이아웃·렌더링·네비게이션
- 사용자 인증·온보딩, 개인화 추천, 검색
- AI 음성(TTS)·푸시 알림
- 편향 분석, 트렌드 키워드 집계
- 관리자 사용자 계정 관리
- 뉴스 출처 등록·수정·삭제 관리 UI/API
- 동시 사용자 1,000명 이상 동시성 대응 (캐시·CDN·수평 확장은 별도 성장 사이클에서 대응)
