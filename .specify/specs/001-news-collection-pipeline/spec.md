# Feature Specification: 뉴스 수집·큐레이션 파이프라인과 카테고리 피드

**Feature Branch**: `001-news-collection-pipeline`

**Created**: 2026-06-08

**Status**: Draft

**Input**: User description: "Newsift의 뉴스 수집·큐레이션 파이프라인과 카테고리 피드 기능"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 자동 뉴스 수집 및 중복 제거 (Priority: P1)

시스템은 등록된 뉴스 출처들을 주기적으로 폴링하여 새 기사를 수집한다.
이미 수집된 기사(동일 소스 + 원문 ID/URL)는 건너뛰고, 처음 보는 기사만 저장한다.
출처 중 일부가 응답하지 않거나 오류를 반환해도 나머지 출처의 수집은 계속 진행한다.

**Why this priority**: 수집 파이프라인이 없으면 피드에 표시할 기사가 없다.
전체 서비스의 데이터 공급 기반이 되며, 중복 방지는 피드 품질의 핵심이다.

**Independent Test**: 수집 스케줄러를 한 번 실행한 뒤, 특정 기사가 정확히 한 번만
저장되었음을 확인하고, 특정 출처를 오프라인으로 두었을 때 나머지 출처의 기사가 정상
저장됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 등록된 뉴스 출처에 새 기사 N건이 게시되어 있을 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 해당 N건이 시스템에 저장되고, 각 기사에 수집 시각이 기록된다.
2. **Given** 이미 수집된 기사와 동일한 URL의 기사가 출처에 다시 노출될 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 기존 기사가 중복 저장되지 않고 건너뛴다.
3. **Given** 출처 A는 정상, 출처 B는 타임아웃 상태일 때,
   **When** 수집 스케줄러가 실행되면,
   **Then** 출처 A의 기사는 정상 수집되고, 출처 B의 실패가 로그에 기록되며 전체 수집 작업은 완료 상태로 종료된다.

---

### User Story 2 - 카테고리 분류 및 AI 요약 생성 (Priority: P2)

새로 수집된 기사에 대해 시스템이 카테고리를 자동 분류하고 AI 요약을 생성한다.
요약은 핵심만(brief), 균형(balanced), 깊이(deep) 세 가지 깊이로 제공된다.
요약 생성에 실패한 기사는 요약 없음(PENDING/FAILED) 상태로 표시되어 피드에는
나타나지만 요약이 없음이 명확히 구분된다.
생성된 요약에는 생성 시각과 AI 생성임을 나타내는 메타정보가 포함된다.

**Why this priority**: 카테고리와 요약은 앱 사용자가 기사를 빠르게 판단하는 핵심 가치다.
수집(P1) 이후 즉시 처리되어야 피드에서 의미 있는 정보를 제공할 수 있다.

**Independent Test**: 수집된 기사 1건에 대해 분류·요약 처리를 실행한 뒤,
카테고리가 할당되고 세 가지 깊이의 요약이 모두 저장되었음을 확인한다.
AI 서비스가 오류를 반환할 때 해당 기사가 FAILED 상태로 저장됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 새로 수집된 기사(카테고리·요약 미할당)가 있을 때,
   **When** 분류·요약 처리가 실행되면,
   **Then** 기사에 카테고리 1개가 부여되고, brief/balanced/deep 요약이 각각 저장되며, 각 요약에 생성 시각이 기록된다.
2. **Given** AI 요약 서비스가 일시적으로 오류를 반환할 때,
   **When** 요약 생성을 시도하면,
   **Then** 해당 기사의 요약 상태가 FAILED로 기록되고, 기사 자체는 카테고리와 함께 피드에 노출된다.
3. **Given** 카테고리 분류 결과가 없거나 모호할 때,
   **When** 분류 처리가 실행되면,
   **Then** 기사에 "기타(OTHER)" 카테고리가 부여된다.

### User Story 3 - 앱 사용자의 카테고리 피드 조회 (Priority: P3)

앱 사용자(인증된 사용자)는 최신순 뉴스 목록 또는 특정 카테고리로 필터링한 뉴스 목록을 조회한다.
각 기사 항목에는 제목, 카테고리, 출처, 발행 시각, brief 요약이 포함된다.
기사 상세 조회 시 원문 링크, 기자, 수집 시각, 전체 요약(선택한 깊이), AI 생성 면책 안내,
요약 생성 시각이 제공된다.
페이지네이션을 통해 대량의 기사를 안정적으로 조회한다.

**Why this priority**: 피드 조회는 앱 사용자가 직접 소비하는 핵심 기능으로,
수집(P1)·분류/요약(P2) 이후에 의미 있는 데이터를 제공할 수 있다.

**Independent Test**: API를 통해 카테고리 필터 없이 피드를 조회했을 때 최신순으로
기사 목록이 반환되는지, 특정 카테고리로 필터링 시 해당 카테고리 기사만 반환되는지 확인한다.
기사 상세 API 호출 시 원문 링크와 요약이 포함되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 여러 카테고리의 기사가 저장되어 있을 때,
   **When** 인증된 사용자가 카테고리 필터 없이 피드를 조회하면,
   **Then** 기사가 원 출처 발행 시각 기준 최신순으로 반환되고, 각 항목에 제목·카테고리·출처·발행 시각·brief 요약이 포함된다.
2. **Given** 경제·금융 카테고리 기사 5건이 저장되어 있을 때,
   **When** 인증된 사용자가 "경제·금융" 카테고리로 피드를 조회하면,
   **Then** 해당 카테고리 기사 5건만 반환된다.
3. **Given** 기사 ID가 유효할 때,
   **When** 인증된 사용자가 기사 상세를 조회하면,
   **Then** 응답에 원문 링크·기자·수집 시각·세 가지 깊이의 요약(상태 포함)·AI 생성 면책 안내·요약 생성 시각이 포함된다.
4. **Given** 요약 생성이 실패한 기사를 조회할 때,
   **When** 사용자가 해당 기사 상세를 조회하면,
   **Then** 요약 상태가 FAILED로 반환되고, 원문 링크가 함께 제공된다.

### User Story 4 - 관리자 파이프라인 모니터링 (Priority: P4)

관리자(인증된 관리자 역할)는 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율,
중복 제거율, 파이프라인 단계별 처리 현황을 API로 조회한다.
이 데이터는 백오피스 UI가 소비하는 API로 제공된다.

**Why this priority**: 운영 모니터링은 서비스 품질 유지에 필수이나,
사용자 기능(P1-P3) 완성 후 제공해도 서비스 운영에 지장이 없다.

**Independent Test**: 관리자 권한 토큰으로 대시보드 API를 호출하여
오늘 수집 건수·요약 완료율·중복 제거율이 실제 저장된 데이터와 일치하는지 확인한다.
일반 사용자 권한으로 호출 시 접근 거부됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 오늘 기사 100건이 수집되고 그 중 80건 요약 완료, 5건 중복 제거된 상태일 때,
   **When** 관리자가 대시보드 API를 조회하면,
   **Then** 수집 100건, 요약 완료율 80%, 중복 제거 5건, 카테고리별 기사 수가 정확히 반환된다.
2. **Given** 일반 사용자 토큰으로 관리자 API를 호출할 때,
   **When** 대시보드 엔드포인트에 요청을 보내면,
   **Then** 접근 거부(403) 응답이 반환된다.

---

### Edge Cases

- 뉴스 출처가 0건 등록된 상태에서 수집 스케줄러가 실행될 때: 수집 작업이 정상 완료되고 처리 건수 0으로 기록된다.
- 특정 출처가 연속으로 실패할 때: 해당 출처의 오류 횟수가 누적 기록되며, 다른 출처 수집에 영향을 주지 않는다.
- AI 요약 서비스 전체 장애 시: 새로 수집된 기사는 카테고리만 부여된 채 PENDING 상태로 저장되고, 서비스 복구 후 재처리 대상이 된다.
- 기사 원문이 삭제되거나 접근 불가능해진 경우: 시스템은 수집 시점에 저장된 메타데이터(제목·발행 시각·출처)와 원문 링크를 유지하고, 원문 재수집을 시도하지 않는다.
- 동일 기사가 두 출처에서 동시에 수집될 때: URL 기준 dedup 처리로 하나만 저장된다.
- 피드 조회 시 조회 결과가 0건인 경우: 빈 목록과 함께 정상 응답(200)이 반환된다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 환경변수로 설정된 주기에 따라 등록된 모든 활성 뉴스 출처에서 기사를 수집해야 한다.
- **FR-002**: 시스템은 기사의 소스 + 원문 URL을 식별 키로 사용해 중복 수집을 방지해야 한다.
- **FR-003**: 시스템은 특정 출처 수집이 실패해도 나머지 출처 수집을 계속 진행해야 한다.
- **FR-004**: 시스템은 각 기사에 원 출처의 발행 시각(published_at)과 시스템의 수집 시각(collected_at)을 구분하여 저장해야 한다.
- **FR-005**: 시스템은 수집된 기사를 경제·금융, 과학, 정치, 스포츠, 세계, 연예·문화, 건강·의학, 자동차, IT, 기타 중 하나로 분류해야 한다.
- **FR-006**: 시스템은 분류된 기사에 대해 brief/balanced/deep 세 가지 깊이의 AI 요약을 생성하고, 각 요약에 생성 시각과 AI 생성 식별 메타정보를 저장해야 한다.
- **FR-007**: 시스템은 요약 생성이 실패한 기사를 FAILED 상태로 구분하여 저장하고, 요약 없이 피드에 노출해야 한다.
- **FR-008**: 시스템은 원문 전체를 저장하거나 재배포하지 않고, 제목·요약·원문 링크·메타정보만 제공해야 한다.
- **FR-009**: 인증된 사용자는 기사 목록을 발행 시각 기준 최신순으로 조회할 수 있어야 하며, 카테고리로 필터링할 수 있어야 한다.
- **FR-010**: 인증된 사용자는 기사 상세(요약, 원문 링크, 출처, 기자, 발행/수집 시각, AI 면책 안내)를 조회할 수 있어야 한다.
- **FR-011**: 기사 목록 조회는 페이지네이션을 지원해야 한다.
- **FR-012**: 인증된 관리자는 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율, 중복 제거율, 파이프라인 단계별 처리 현황을 조회할 수 있어야 한다.
- **FR-013**: 관리자 모니터링 API는 관리자 역할을 가진 사용자만 접근 가능해야 한다.
- **FR-014**: 수집·분류·요약 파이프라인 각 단계의 시작/종료/실패 이벤트가 관측 가능하게 기록되어야 한다.

### Key Entities *(include if feature involves data)*

- **Source (뉴스 출처)**: 기사를 수집할 외부 뉴스 제공처. 이름, 피드 URL, 활성 여부, 마지막 수집 시각, 연속 실패 횟수를 포함한다.
- **Article (기사)**: 수집된 개별 뉴스 기사. 제목, 원문 링크(dedup 식별 키), 출처 참조, 기자, 발행 시각(published_at), 수집 시각(collected_at), 카테고리, 요약 상태(PENDING/COMPLETED/FAILED)를 포함한다.
- **Summary (요약)**: 기사에 귀속되는 AI 생성 요약. 깊이(BRIEF/BALANCED/DEEP), 요약 본문, 생성 시각, AI 생성 식별 플래그를 포함한다. 기사당 최대 3개(깊이별 1개).
- **Category (카테고리)**: 기사의 주제 분류값. 고정된 열거형(경제·금융, 과학, 정치, 스포츠, 세계, 연예·문화, 건강·의학, 자동차, IT, 기타).
- **PipelineStats (파이프라인 통계)**: 관리자 대시보드를 위한 집계 뷰. 오늘 수집 건수, 카테고리별 기사 수, 요약 완료율, 중복 제거 건수.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 출처에 새 기사가 게시된 후 시스템의 수집 주기 내에 카테고리 피드에서 해당 기사가 조회된다.
- **SC-002**: 동일 기사가 피드에 중복으로 노출되지 않는다 (중복 노출 건수 = 0).
- **SC-003**: 특정 출처에 장애가 발생한 상태에서도 피드 조회가 정상 동작하고 사용자에게 기사 목록이 반환된다.
- **SC-004**: 수집된 모든 기사에 카테고리가 부여된다 (카테고리 미할당 기사 비율 = 0%).
- **SC-005**: 요약 생성이 실패한 기사는 요약 상태(FAILED)가 명시된 채 피드와 상세 조회에서 확인 가능하다.
- **SC-006**: 관리자는 오늘 수집 건수·요약 완료율·중복 제거율·카테고리별 기사 수를 단일 API 호출로 확인할 수 있다.
- **SC-007**: 수집·분류·요약 파이프라인의 각 단계별 처리 결과가 로그에서 추적 가능하다.

## Assumptions

- 뉴스 출처 등록·관리(Source CRUD)는 이번 사이클 범위 밖이며, 출처는 초기 데이터 또는 별도 관리 기능으로 사전 등록되어 있다고 가정한다.
- 사용자 인증 및 역할(일반 사용자 / 관리자) 관리는 별도 명세에서 다루며, 이 기능은 인증 시스템이 이미 동작 중임을 전제한다.
- 카테고리 분류와 AI 요약은 외부 AI 서비스를 통해 처리된다. AI 서비스 선택·프롬프트 설계는 구현 단계에서 결정한다.
- 수집 주기는 환경변수로 관리되며, 기본값은 합의를 통해 결정한다. 이 명세에서 구체적인 수치(예: 15분)를 고정하지 않는다.
- 기사 피드 조회는 인증된 사용자만 접근 가능하다. 공개(비인증) 피드는 이번 사이클 범위 밖이다.
- AI 요약 재처리(FAILED → 재시도) 스케줄은 이번 사이클에서 "수동 또는 다음 수집 주기 시 함께 처리"로 단순화한다.
- 기사 발행 시각(published_at)은 외부 소스가 제공하는 값을 사용하며, 누락된 경우 수집 시각으로 대체한다.

## Out of Scope

- 클라이언트(React/Android) UI 레이아웃·렌더링·네비게이션
- 사용자 인증·온보딩, 개인화 추천, 검색
- AI 음성(TTS)·푸시 알림
- 편향 분석, 트렌드 키워드 집계
- 관리자 사용자 계정 관리
- 뉴스 출처 등록·수정·삭제 관리 UI/API
