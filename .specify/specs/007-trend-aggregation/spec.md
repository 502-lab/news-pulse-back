# Feature Specification: 트렌드 집계 엔진

**Feature Branch**: `007-trend-aggregation`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "시간대·카테고리별 트렌드(급상승 키워드·이슈), Top5, WoW 급상승, 이슈 클러스터링(co-occurrence MVP, IssueClusterer 격리)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 지금 뜨는 키워드 Top5 (Priority: P1)

독자가 홈/탐색 화면에서 "지금 뜨는 키워드 Top5"를 확인한다(W-U-01 / A-06). 최근 트렌드 윈도우 기준으로 노출 빈도가 높고 급상승 중인 키워드 상위 5개를, 각 키워드의 노출 횟수(count)와 증감률(deltaPct)과 함께 본다.

**Why this priority**: 트렌드 기능의 핵심 진입점이자 가장 가시적인 사용자 가치. 단독으로도 "지금 무슨 이슈가 뜨는가"를 전달한다.

**Independent Test**: 트렌드 슬롯 데이터가 존재하는 상태에서 Top5 조회 시 키워드 5개와 각 count·deltaPct가 내림차순으로 반환되는지 확인. 데이터가 없으면 빈 목록.

**Acceptance Scenarios**:

1. **Given** 최근 윈도우에 키워드 집계가 존재, **When** Top5 조회, **Then** 급상승 점수 상위 5개 키워드와 `term`·`count`·`deltaPct` 반환
2. **Given** 카테고리 필터 지정, **When** 해당 카테고리 Top5 조회, **Then** 그 카테고리 기사에서 집계된 키워드만 반환
3. **Given** 트렌드 집계 데이터가 아직 없음, **When** Top5 조회, **Then** 빈 목록(에러 아님)

---

### User Story 2 — 트렌드 집계 파이프라인 (Priority: P1)

시스템이 수집된 기사(001 시계열)에서 키워드를 추출하고, 시간대(슬롯)·카테고리별로 키워드 출현 빈도를 주기적으로 집계한다. 집계 결과는 윈도우 단위로 누적·보존되어 Top5·히트맵·WoW 조회의 기반 데이터가 된다.

**Why this priority**: 키워드/이슈 데이터를 생산하는 인프라. US1·US3·US4·US5의 선행 조건.

**Independent Test**: 특정 시간대에 수집된 기사들에 대해 집계 작업 실행 후, 해당 슬롯·카테고리의 키워드 count가 DB에 저장되는지 확인.

**Acceptance Scenarios**:

1. **Given** 슬롯 구간 내 수집 기사 N건, **When** 집계 스케줄러 실행, **Then** 각 키워드별 출현 count가 (슬롯, 카테고리) 단위로 저장됨
2. **Given** 동일 슬롯에 대해 집계가 재실행됨, **When** 재집계, **Then** 중복 누적 없이 멱등하게 갱신(기사+키워드에서 재산출 가능)
3. **Given** 보존 기간을 초과한 슬롯, **When** 보존 정리 실행, **Then** 만료 슬롯이 제거되고 최신 데이터에는 영향 없음

---

### User Story 3 — 히트맵·워드클라우드 데이터 (Priority: P2)

독자가 트렌드 화면(W-02)에서 시간대·카테고리별 키워드 분포를 히트맵과 워드클라우드로 본다. 시스템은 (시간 슬롯 × 카테고리) 격자의 키워드 강도와, 가중치(count/deltaPct)를 가진 키워드 목록을 제공한다.

**Why this priority**: 트렌드의 시각적 탐색 경험. Top5보다 넓은 분포를 보여주나, Top5(P1) 없이도 독립적으로 가치 있는 부가 뷰.

**Independent Test**: 다수 슬롯·카테고리에 집계가 존재하는 상태에서 히트맵 데이터 조회 시 (슬롯, 카테고리, 강도) 격자가, 워드클라우드 조회 시 (term, weight) 목록이 반환되는지 확인.

**Acceptance Scenarios**:

1. **Given** 여러 슬롯·카테고리 집계 존재, **When** 히트맵 데이터 조회, **Then** 시간 슬롯 × 카테고리별 키워드 강도 격자 반환
2. **Given** 특정 윈도우, **When** 워드클라우드 데이터 조회, **Then** 키워드별 가중치(`term`, `weight`) 목록 반환
3. **Given** 빈 윈도우, **When** 조회, **Then** 빈 격자/목록 반환

---

### User Story 4 — WoW(주간 대비) 급상승 (Priority: P2)

독자가 이번 주 대비 지난주에 비해 급상승한 키워드를 확인한다. 시스템은 현재 주간 윈도우의 키워드 count를 직전 주간과 비교해 증감률(deltaPct)을 산출하고, 신규(직전 주 0건) 또는 급상승 키워드를 식별한다.

**Why this priority**: "최근 떴다"를 넘어 "추세적으로 떠오른다"를 보여주는 상위 분석. US2 데이터 누적이 1주 이상 쌓여야 의미가 생긴다.

**Independent Test**: 두 주간 윈도우에 서로 다른 키워드 분포가 존재하는 상태에서 WoW 조회 시 deltaPct가 정확히 산출되고 급상승 순으로 정렬되는지 확인.

**Acceptance Scenarios**:

1. **Given** 이번 주·지난주 키워드 집계 존재, **When** WoW 급상승 조회, **Then** `deltaPct` 내림차순 키워드 목록 반환
2. **Given** 직전 주 0건이던 키워드가 이번 주 등장, **When** 조회, **Then** 신규 급상승으로 식별(증감률 표기 규칙 적용)
3. **Given** 노이즈 키워드(최소 빈도 미만), **When** 조회, **Then** 결과에서 제외

---

### User Story 5 — 이슈 클러스터링 (Priority: P3)

독자가 개별 키워드를 넘어 "이슈" 단위(연관된 기사 묶음 + 대표 키워드)로 트렌드를 본다. 시스템은 키워드 동시출현(co-occurrence)을 기반으로 관련 기사를 이슈로 묶고, 각 이슈에 증감과 대표 키워드 3개를 부여한다.

**Why this priority**: 키워드 나열을 넘어선 의미 단위 묶음. MVP 클러스터링 품질이 검증돼야 하며, Top5/히트맵 없이도 독립 가치가 있으나 우선순위는 낮다.

**Independent Test**: 동일 슬롯에서 함께 자주 등장한 키워드를 가진 기사들이 하나의 이슈로 묶이고, 이슈에 관련 기사 목록·증감·대표 키워드 3개가 부여되는지 확인.

**Acceptance Scenarios**:

1. **Given** 동시출현 키워드를 공유하는 기사 그룹, **When** 이슈 클러스터링 실행, **Then** 관련 기사 묶음 + 대표 키워드 3개를 가진 Issue 생성
2. **Given** 기사·키워드 데이터, **When** 클러스터러 재실행, **Then** 수작업 큐레이션 없이 기사+키워드에서 이슈가 재산출됨(멱등)
3. **Given** 단독 키워드만 가진 기사, **When** 클러스터링, **Then** 이슈로 묶이지 않거나 단일 기사 이슈로 처리(노이즈 규칙 적용)

---

### Edge Cases

- 동일 키워드가 한 기사에 여러 번 등장 시 기사당 1회로 집계(기사 단위 빈도, 토큰 빈도 아님).
- 슬롯 경계에 걸친 기사는 수집 시각(`first_collected_at`) 기준 단일 슬롯에 귀속.
- deltaPct: 직전 윈도우 count가 0인 신규 키워드는 백분율 무한대 대신 "신규" 플래그/별도 표기로 처리한다(분모 0 회피).
- 최소 빈도(노이즈 컷) 미만 키워드는 Top5·WoW·워드클라우드에서 제외한다.
- 불용어(조사·일반어 등) 제거: 의미 없는 고빈도 토큰이 Top5를 점유하지 않도록 한다.
- 카테고리 미분류(분류 실패) 기사: 카테고리별 집계에서는 "기타"로 귀속, 전체 집계에는 포함.
- 이슈 클러스터링 결과가 비어도(동시출현 부족) 키워드 트렌드(US1/3/4)는 정상 동작해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 수집된 기사(제목 + AI 요약 `summaries.content`, 요약 없으면 제목만 fallback)에서 한국어 형태소 분석(명사 NNG/NNP 추출)으로 키워드를 추출해야 한다. 형태소 분석기는 Lucene Nori(JVM 라이브러리, 네이티브 의존 없음)를 신규 도입한다. 추출 로직은 `KeywordExtractor` 포트 뒤에 격리하여 교체 가능하게 한다(004 TtsProvider / 005 PushNotificationPort 패턴). 추출 품질은 런타임 eval로 확인한다.
- **FR-002**: 시스템은 추출된 키워드를 (시간 슬롯, 카테고리) 단위로 기사당 1회 빈도로 집계해야 한다.
- **FR-003**: 집계는 멱등해야 한다 — 동일 슬롯 재집계 시 중복 누적 없이 기사+키워드에서 재산출 가능해야 한다(수작업 큐레이션 금지).
- **FR-004**: 시스템은 최근 윈도우 기준 급상승 키워드 Top5를 `term`·`count`·`deltaPct`와 함께 제공하는 API를 제공해야 한다(카테고리 필터 지원).
- **FR-005**: 시스템은 키워드별 `term`, `count`, `deltaPct`, `slots[]`(시간대별 시계열)를 제공해야 한다.
- **FR-006**: 시스템은 시간 슬롯 × 카테고리 격자의 히트맵 데이터와, 가중치를 가진 워드클라우드 데이터(W-02)를 제공하는 API를 제공해야 한다.
- **FR-007**: 시스템은 WoW(이번 주 vs 지난주) 증감률을 산출하고 급상승 키워드 목록을 제공해야 한다. 직전 윈도우 0건 키워드는 "신규"로 식별한다.
- **FR-008**: 시스템은 최소 빈도(노이즈 컷)와 불용어 제거 규칙을 적용하여 의미 없는 키워드를 트렌드 결과에서 제외해야 한다.
- **FR-009**: 시스템은 트렌드 슬롯 데이터의 윈도우·보존 기간을 정의하고, 보존 초과 데이터를 주기적으로 정리해야 한다.
- **FR-010**: 시스템은 키워드 동시출현(co-occurrence) 기반으로 관련 기사를 이슈(Issue)로 묶고, 각 이슈에 관련 기사 묶음·증감·대표 키워드 3개를 부여해야 한다.
- **FR-011**: 이슈 클러스터링 로직은 `IssueClusterer` 인터페이스(포트) 뒤에 격리되어, 트렌드 집계·데이터 모델·API를 변경하지 않고 구현을 교체할 수 있어야 한다(004 TtsProvider / 005 PushNotificationPort와 동일한 격리 패턴).
- **FR-012**: 이슈는 기사 + 키워드 데이터에서 재산출 가능해야 한다 — 클러스터러 교체 시 재클러스터링만으로 전환할 수 있어야 한다.
- **FR-013**: 트렌드 read 엔드포인트(Top5/히트맵/WoW)는 공개(permitAll)여야 한다. SecurityConfig에 명시적 permitAll 규칙과 정당화 코멘트를 둔다(Constitution §VI: 공개 엔드포인트는 명시적으로 선언). 비민감 전역 집계이므로 인증을 요구하지 않는다. read는 배치 집계가 미리 산출·저장한 결과를 서빙하며 on-the-fly 재계산을 하지 않는다.
- **FR-014**: 트렌드 집계 스케줄러는 시작·종료·처리 건수·실패 건수를 구조적 로그로 기록해야 한다(민감 정보 제외).
- **FR-015**: 공개 트렌드 read는 캐싱·rate-limit을 적용한다(비기능 요구). 단 Redis는 도입하지 않는다 — 저장된 배치 집계 + 인덱스 기반 read를 1순위로 하며, 규모 증가 시 materialized view로 전환한다.

### Key Entities *(include if feature involves data)*

- **TrendKeyword**: 키워드 트렌드 집계 단위 — `term`, `category`, `slot`(시간 버킷), `count`(기사당 1회 빈도), 파생값 `deltaPct`(직전 윈도우 대비). (슬롯 × 카테고리 × term) 단위로 존재.
- **KeywordView**: 조회용 키워드 집계 — `term`, `count`(윈도우 합), `deltaPct`, `slots[]`(시간대별 시계열). API 응답 형태.
- **Issue**: 이슈 단위 묶음 — 관련 기사 묶음(article 참조), 증감, 대표 키워드 3개. 기사+키워드에서 재산출 가능. (선택) `clusteringMethod`(어떤 방식으로 산출됐는지 — A: co-occurrence / C: embedding 혼재 추적용; plan에서 결정).
- **IssueClusterer**: 이슈 산출 포트(개념적 인터페이스). MVP 구현은 co-occurrence 기반. 교체 가능.
- **KeywordExtractor**: 키워드 추출 포트(개념적 인터페이스). MVP 구현은 Lucene Nori 명사(NNG/NNP) 추출. 교체 가능(향후 다른 분석기/사전).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 독자가 홈/탐색에서 "지금 뜨는 키워드 Top5"를 즉시(체감 1초 이내) 확인할 수 있다.
- **SC-002**: 트렌드 Top5/히트맵/WoW 조회 응답이 일관되게 빠르다(p95 ≤ 3s). [TODO: 측정조건·동시요청 수 확정]
- **SC-003**: 최근 윈도우에 수집 기사가 존재하면, 집계 주기 1회 경과 후 Top5에 키워드가 반영된다(반영 지연 ≤ 집계 주기 + 여유).
- **SC-004**: WoW 급상승 목록의 증감률이 실제 두 주간 집계와 일치하며, 신규 키워드(직전 0건)가 분모 0 없이 정확히 식별된다.
- **SC-005**: 이슈 클러스터러를 교체해도 트렌드 집계·데이터 모델·조회 API는 변경 없이 동작하며, 이슈는 재클러스터링만으로 전환된다.
- **SC-006**: 트렌드 결과에 불용어·노이즈 키워드(최소 빈도 미만)가 노출되지 않는다.

## Assumptions

- 키워드 트렌드는 001에서 수집·저장된 기사 시계열(`first_collected_at` 기준)에 의존한다. 별도 외부 트렌드 소스는 사용하지 않는다.
- 실시간 트렌드 윈도우/슬롯: 기본 슬롯은 시간 단위(hourly), 실시간 Top5는 최근 24시간 윈도우, WoW는 주간 윈도우를 가정한다(구체값은 plan/clarify에서 확정).
- 트렌드 슬롯 데이터 보존: 기본 90일(001 기사 보존과 정합). 보존 초과 슬롯은 주기적으로 정리한다.
- 키워드 빈도는 "기사 단위"(기사당 1회)로 집계한다(토큰 빈도 아님) — 한 기사 내 반복이 트렌드를 왜곡하지 않도록.
- 이슈 클러스터링 MVP는 키워드 동시출현(co-occurrence) 방식이다. 이슈는 수작업 큐레이션이 아니라 기사+키워드에서 재산출된다.
- **범위 밖**: 관리자 트렌드 운영 뷰(008), rising/급상승 알림 발송(005 알림 시스템이 트렌드 데이터를 소비; 본 스펙은 데이터 제공만), 기사 비교(006 US5에서 분리된 별도 후속 스펙).
- **v2 업그레이드 경로(범위 밖, 미래 참조용)**: co-occurrence 클러스터 품질이 부족해지면 임베딩 기반 클러스터링으로 업그레이드한다. 고려사항 — 임베딩 소스(예: Gemini embeddings), 벡터 저장, 코사인 유사도 클러스터링(DBSCAN 등), 임베딩 호출 비용·지연. `IssueClusterer`에 `EmbeddingIssueClusterer` 구현을 추가해 교체하는 형태이며, 상세는 별도 후속 스펙/ADR에서 다룬다. 전환기 A/C 혼재 대비로 `Issue.clusteringMethod` 필드를 둘지 plan에서 결정한다.
- 카테고리는 001/002의 기존 카테고리 분류를 재사용한다(별도 분류 체계 신설 안 함). 분류 실패 기사는 카테고리별 집계에서 "기타"로 귀속한다.
- 키워드 추출은 Lucene Nori(JVM, 네이티브 의존 없음)로 신규 도입하며 `KeywordExtractor` 포트 뒤에 격리한다. 기존 스택에 형태소 분석기·키워드 추출 자산이 없음을 코드 확인함(pg_bigm은 substring 검색이라 키워드 추출 아님, FollowKeyword는 사용자 팔로우 키워드, 006 rationale_keywords는 편향 근거로 트렌드 목적과 다름).
- 트렌드 read 엔드포인트는 공개(permitAll)다. **공개→회원전용 전환은 비파괴 변경이 아니다(breaking)** — 향후 게이팅이 필요하면 별도 고려(버전/마이그레이션 경로)한다.
- 트렌드 read 성능은 Redis 없이 저장된 배치 집계 + 인덱스 read로 충족하며, 규모 증가 시 materialized view로 전환한다(006 OutletBias와 동일 방향).
