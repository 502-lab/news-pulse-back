## Clarifications

### Session 2026-06-23

- Q: 스케줄러 토글/주기 영속성(FR-031) — 인메모리 플래그 vs DB 영속? → A: **SchedulerSetting DB 영속**. admin이 끈 스케줄러는 앱 재기동 후에도 비활성 유지(007 단일 인스턴스 전제). 신규 엔티티 `SchedulerSetting` 확정.
- Q: 에러 로그 출처(FR-051) — 신규 에러 캡처 인프라 vs 기존 FAILED 상태 집계? → A: **기존 FAILED 상태 집계**(요약 `summary_status` FAILED·편향 status FAILED·알림 아웃박스 FAILED). 전용 에러 저장소 미도입(cross-cutting 캡처는 008 밖). `AdminErrorLog` 엔티티 불필요.
- Q: AdminAuditLog(FR-060) — 008 포함 여부·감사 범위? → A: **포함, 변형(부수효과) 액션만 감사**(역할 변경·계정 활성/비활성·기사 숨김·스케줄러 제어·공지 CRUD·푸시 발송). 단순 조회 제외. 신규 엔티티 `AdminAuditLog` 확정(`AdminTargetType` 재활용).
- Q: 기사 영구 삭제(FR-034) — 영구 삭제 필요 vs 숨김만? → A: **숨김(소프트)만, 영구 삭제 미제공**. `articles(id)` 참조 FK가 전부 `ON DELETE CASCADE`라 영구 삭제 시 `article_keyword`·`bias_analysis`·`summaries`·`saved_articles`가 비가역 연쇄 삭제(트렌드·편향 집계 소스 손실) → 위험 원천 차단. 영구 삭제(법적/저작권 takedown)는 가드·감사 설계 후 후속 피처.
- Q: 자기보호 가드(FR-014) — 자기 자신 / 마지막 ADMIN? → A: **둘 다 적용**. (a) 관리자 자기 자신의 역할 강등·비활성화 금지, (b) 마지막 ADMIN의 강등·비활성화 금지.

## User Scenarios & Testing *(mandatory)*

> 모든 스토리는 운영자(관리자, `ADMIN` 역할)가 actor다. 일반 사용자(`USER`)는 어떤 스토리에도 접근할 수 없다.
> 본 기능은 **기존 데이터·파이프라인 위의 운영(operations) 레이어**이며, 신규 사용자-facing 기능이 아니다.

### User Story 1 - 사용자 관리 (Priority: P1)

운영자가 가입 사용자 전체를 조회·검색하고, 개별 사용자의 상태(활성/비활성)와 역할(USER↔ADMIN)을 변경해
악용 계정을 차단하거나 운영자를 추가/회수한다.

**Why this priority**: 운영의 가장 기본이자 위험 통제의 핵심. 악용 계정 차단·운영자 권한 관리는 다른 어떤
모니터링·콘텐츠 제어보다 선행되어야 하는 최소 운영 능력이다. accounts 테이블과 002 인가가 이미 있어
가장 적은 신규 인프라로 즉시 가치를 낸다.

**Independent Test**: 관리자 토큰으로 사용자 목록을 조회하고, 한 사용자의 역할을 USER→ADMIN으로 변경한 뒤
다시 USER로 되돌리고, 한 사용자를 비활성화→재활성화한다. 일반 사용자 토큰으로 동일 호출 시 거부됨을 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자로 로그인, **When** 사용자 목록을 페이지네이션·검색(이메일/상태/역할 필터)으로 조회, **Then** 각 사용자의 식별자·이메일·역할·상태·가입일·가입유형이 반환된다.
2. **Given** 한 USER 사용자, **When** 관리자가 역할을 ADMIN으로 변경, **Then** 해당 계정의 역할이 ADMIN으로 바뀌고 이후 관리자 기능에 접근 가능해진다.
3. **Given** 활성 사용자, **When** 관리자가 비활성화, **Then** 해당 계정 상태가 비활성으로 바뀌고 그 사용자의 로그인/토큰 발급이 차단된다.
4. **Given** 일반 USER 토큰, **When** 사용자 관리 API 호출, **Then** 403으로 거부된다.
5. **Given** 시스템에 관리자가 1명뿐, **When** 그 관리자가 자기 자신을 USER로 강등 시도, **Then** 시스템이 거부한다(관리자 0명 lockout 방지).

### User Story 2 - 핵심 운영 모니터링 조회 (Priority: P1)

운영자가 단일 대시보드에서 서비스 핵심 지표(KPI)와 파이프라인 상태를 한눈에 본다: 수집·요약·편향(006)·
트렌드(007) 처리 현황, 소스별 수집량(source_daily_usage 재활용), 스케줄러 상태, 활성 사용자 수 등.

**Why this priority**: "지금 서비스가 정상인가"를 답하는 읽기 전용 가시성. 제어(US3)·발송(US4)보다 먼저
관측이 가능해야 운영 의사결정이 선다. 전부 기존 테이블 집계라 신규 쓰기 인프라가 없어 P1로 함께 출시 가능.

**Independent Test**: 관리자 토큰으로 KPI 요약·파이프라인 stats·소스 수집량·스케줄러 상태 엔드포인트를
호출해 기존 데이터 기반 집계 값이 반환되는지 확인한다. 데이터가 없을 때 0/빈 값으로 안전하게 응답하는지 본다.

**Acceptance Scenarios**:

1. **Given** 관리자, **When** KPI 요약 조회, **Then** 총/활성 사용자 수, 기간 내 수집 기사 수, 요약 완료율, 편향 분석 완료율, 트렌드 이슈 수 등 핵심 지표가 반환된다.
2. **Given** 관리자, **When** 파이프라인 stats 조회, **Then** 처리 단계별 상태 분포(예: 요약 PENDING/COMPLETED/FAILED, 편향 PENDING/DONE/FAILED) 카운트가 반환된다.
3. **Given** 관리자, **When** 소스별 수집량 조회(기간 지정), **Then** source_daily_usage 기반 소스·일자별 call_count 집계가 반환된다.
4. **Given** 관리자, **When** 스케줄러 상태 조회, **Then** 각 스케줄러(수집·요약·편향·트렌드·알림 등)의 활성 여부·최근 실행 시각·다음 예정/주기가 반환된다.
5. **Given** 분석 데이터가 전혀 없는 신규 환경, **When** 모니터링 조회, **Then** 오류 없이 0·빈 목록으로 응답한다.

### User Story 3 - 수집·콘텐츠 운영 제어 (Priority: P2)

운영자가 런타임에 수집/처리 파이프라인을 제어한다: 스케줄러 수동 실행, 스케줄러 on/off 토글(영속),
수집 제외 키워드 관리, 실패한 요약의 재시도 트리거, 부적절 기사 숨김(가역). (주기 동적 변경·영구 삭제는 후속.)

**Why this priority**: 관측(US2) 이후 필요한 능동 개입. 장애·악성 콘텐츠 대응에 중요하나, 읽기 모니터링과
사용자 관리가 선행되어야 안전하게 쓸 수 있다. 일부 제어는 신규 제어용 저장소가 필요해 P1보다 뒤.

**Independent Test**: 관리자가 수집 스케줄러를 수동 실행시키고, 제외 키워드를 추가/삭제하고, 한 기사를
숨김 처리한 뒤 일반 피드에서 노출되지 않음을 확인한다. 실패 요약 1건에 재시도를 걸어 재처리 큐에 들어감을 본다.

**Acceptance Scenarios**:

1. **Given** 관리자, **When** 특정 스케줄러 수동 실행 요청, **Then** 해당 작업이 1회 즉시 트리거되고 실행 결과(시작/완료/오류)가 보고된다.
2. **Given** 관리자, **When** 스케줄러를 비활성화, **Then** 해당 스케줄러의 자동 실행이 멈추고 모니터링(US2)에 비활성으로 반영된다.
3. **Given** 관리자, **When** 제외 키워드 "광고" 추가, **Then** 이후 트렌드/수집 집계에서 해당 키워드가 배제된다.
4. **Given** 요약이 FAILED인 기사, **When** 관리자가 재시도 트리거, **Then** 해당 기사의 요약이 재처리 대상으로 전환된다.
5. **Given** 부적절 기사 1건, **When** 관리자가 숨김 처리, **Then** 그 기사는 일반 사용자 피드·검색·트렌드·상세(404)에서 제외되되 데이터는 보존되고 unhide로 가역이다(영구 삭제는 미제공 — 후속).

### User Story 4 - 공지 관리 + 어드민 푸시 발송 (Priority: P2)

운영자가 서비스 공지(Notice)를 작성·수정·게시·내림(CRUD)하고, 필요 시 005 알림 파이프라인을 통해
사용자에게 어드민 푸시/인앱 알림을 발송한다.

**Why this priority**: 사용자 커뮤니케이션 능력. 운영에 유용하지만 서비스 정상성(US1·US2)보다 후순위.
공지는 신규 Notice 엔티티가 필요하고, 푸시는 기존 005 파이프라인 호출이라 중간 복잡도.

**Independent Test**: 관리자가 공지를 생성→게시→수정→내림하고, 게시된 공지가 공개 조회에 노출됨을 확인한다.
어드민 푸시를 발송해 005 알림 아웃박스에 적재되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자, **When** 공지 생성(제목·본문·게시여부), **Then** 공지가 저장되고 식별자가 반환된다.
2. **Given** 게시된 공지, **When** 사용자/클라이언트가 공개 공지 목록 조회, **Then** 게시 상태인 공지만 노출된다(미게시 초안 제외).
3. **Given** 관리자, **When** 공지 수정·게시중단·삭제, **Then** 변경이 반영되고 비게시 공지는 공개 목록에서 사라진다.
4. **Given** 관리자, **When** 어드민 푸시 발송(대상·제목·본문), **Then** 005 알림 파이프라인에 발송 작업이 적재되고 중복 없이 처리된다.

### User Story 5 - 심층 운영 통계·에러 로그·수집량 상세 (Priority: P3)

운영자가 추가 심층 정보를 본다: 기간별 운영 통계(OpsStats) 추이, 시스템 에러/실패 로그 조회, 소스별
수집량 상세 드릴다운.

**Why this priority**: 있으면 좋은 심화 가시성. 핵심 운영(P1·P2) 없이도 서비스는 운영 가능하므로 마지막.
에러 로그는 현재 저장소가 없어 신규 캡처가 필요할 수 있어 가장 불확실·후순위.

**Independent Test**: 관리자가 기간을 지정해 OpsStats 추이와 수집량 상세를 조회하고, 최근 실패 이벤트가
에러 로그 조회에 나타나는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 관리자, **When** 기간 지정 OpsStats 조회, **Then** 일자별 수집·요약·편향·트렌드 처리량 추이가 반환된다.
2. **Given** 처리 실패가 발생, **When** 관리자가 에러 로그 조회(필터·페이지네이션), **Then** 실패 항목의 시각·유형·대상·원인 요약이 시간 역순으로 반환된다.
3. **Given** 관리자, **When** 특정 소스의 수집량 상세 조회, **Then** 일자별 호출 수·예산 대비 사용률 등 드릴다운이 반환된다.

### Edge Cases

- **관리자 lockout**: 마지막 ADMIN이 자신/타인을 강등·비활성화해 운영자 0명이 되는 상황 → 거부(FR-014 b).
- **자기 계정 강등/비활성화**: 관리자가 본인의 역할 강등 또는 비활성화 시도 → 거부(FR-014 a).
- **기사 숨김 누락 경로**: hidden 기사가 일부 사용자향 읽기 경로(피드·검색·트렌드 입력·북마크·알림)에서 필터 누락 시 "숨겼는데 노출" 버그 → 모든 경로에서 제외 보장 필요(FR-035).
- **hidden 기사 상세/딥링크 접근**: 일반 사용자가 hidden 기사의 상세·딥링크(URL 직접 접근, 북마크했던 기사 포함)에 접근 시 **404(존재 비노출)** 로 응답한다 — 목록 제외만으로는 부족하고 상세 진입까지 차단해야 함. **어드민은 hidden 기사 상세 조회 가능**. (북마크했던 기사가 hidden되면 그 상세도 404 — 노출 차단의 본질.)
- **숨김 가역성**: 숨김(hidden)은 unhide로 가역. 영구 삭제는 미제공(데이터·집계 보존).
- **스케줄러 중복 실행**: 자동 실행 중 수동 실행 트리거 → 동시 실행 경합(단일 인스턴스 가정과 충돌) 방지/거부.
- **스케줄러 토글 영속성**: 비활성화 후 앱 재시작 시 설정이 유지되는가(런타임 메모리 vs 영속 저장).
- **존재하지 않는 사용자/기사 대상 작업**: 404 일관 처리.
- **권한 경계**: 모든 `/api/v1/admin/**`는 ADMIN 외 401(미인증)/403(비ADMIN). 비활성화된 ADMIN 계정의 잔여 토큰 처리.
- **빈 데이터**: 신규 환경에서 모든 모니터링·통계가 0/빈 값으로 안전 응답.
- **공지 본문 검증**: 빈 제목/본문, 과도한 길이, 게시 상태 전이(초안↔게시) 일관성.

## Requirements *(mandatory)*

### Functional Requirements

**공통 / 인가**

- **FR-001**: 시스템은 모든 어드민 기능을 `/api/v1/admin/**` 경로에 두고 `ADMIN` 역할 보유자만 접근을 허용해야 한다(기존 인가 규칙 재활용, 신규 역할 추가 없음).
- **FR-002**: 비인증 요청은 401, 인증되었으나 ADMIN이 아닌 요청은 403으로 거부해야 한다.
- **FR-003**: 시스템은 어드민 기능에서 사용자 식별자·내부 엔티티를 API 경계에서 그대로 노출하지 않고 응답 전용 표현으로 반환해야 한다.

**US1 — 사용자 관리**

- **FR-010**: 시스템은 가입 사용자를 페이지네이션과 필터(이메일 부분검색·상태·역할·가입유형)로 조회할 수 있어야 한다.
- **FR-011**: 시스템은 개별 사용자의 요약 통계(UserStats: 가입일, 마지막 활동/로그인, 저장 기사 수 등 기존 데이터로 산출 가능한 항목)를 제공해야 한다.
- **FR-012**: 관리자는 사용자 역할을 USER↔ADMIN으로 변경할 수 있어야 한다.
- **FR-013**: 관리자는 사용자를 활성/비활성으로 전환할 수 있어야 하며, 비활성 사용자는 로그인·토큰 발급이 차단되어야 한다.
- **FR-014**: 시스템은 (a) 관리자가 **자기 자신**의 역할 강등·비활성화, (b) **마지막 ADMIN**의 강등·비활성화를 모두 거부해야 한다(자기-lockout 및 운영자 0명 상태 동시 방지).

**US2 — 모니터링 조회(읽기 전용)**

- **FR-020**: 시스템은 핵심 KPI 요약(총·활성 사용자 수, 기간 내 수집 기사 수, 요약 완료율, 편향 분석 완료율, 트렌드 이슈 수 등)을 단일 조회로 제공해야 한다.
- **FR-021**: 시스템은 파이프라인 처리 상태 분포(요약·편향 등 단계별 상태 카운트)를 제공해야 한다.
- **FR-022**: 시스템은 소스별 수집량을 기존 `source_daily_usage`(일자별 call_count) 기반으로 집계 제공해야 한다.
- **FR-023**: 시스템은 각 스케줄러의 상태(활성 여부·최근 실행 시각·주기/다음 예정)를 조회 제공해야 한다.
- **FR-024**: 모든 모니터링 조회는 데이터 부재 시 오류 없이 0·빈 값으로 응답해야 한다.
- **FR-025**: 시스템은 편향(006)·트렌드(007)의 어드민 관점 요약 뷰(전역 분포·집계)를 제공해야 한다.

**US3 — 수집·콘텐츠 운영 제어**

- **FR-030**: 관리자는 개별 스케줄러를 수동으로 1회 즉시 실행시킬 수 있어야 하며, 실행 결과가 보고되어야 한다.
- **FR-031**: 관리자는 개별 스케줄러를 **활성/비활성으로 토글**할 수 있어야 하며, 이 설정은 `SchedulerSetting`으로 **DB에 영속**되어 앱 재기동 후에도 유지되어야 한다(끈 스케줄러가 재기동으로 재활성되지 않음). **MVP는 enabled 토글 + 수동 실행(FR-030)만 노출**한다. 주기(interval)의 런타임 동적 변경은 `@Scheduled` 빈 등록 시점 고정 제약으로 **본 범위에서 제외**(스키마 `interval_override_ms`는 두되 API 미노출 — 저장만 하고 적용 안 되는 거짓 약속 회피, 동적 재스케줄은 후속 피처).
- **FR-032**: 관리자는 수집/집계에서 배제할 제외 키워드를 추가·조회·삭제할 수 있어야 하며, 등록된 키워드는 이후 집계에서 배제되어야 한다.
- **FR-033**: 관리자는 실패(FAILED)한 요약을 재시도 대상으로 전환(재처리 트리거)할 수 있어야 한다.
- **FR-034**: 관리자는 기사를 **숨김(hidden, 가역)** 처리·해제(unhide)할 수 있어야 하며, 숨김은 일반 노출만 차단하고 데이터는 보존한다. **영구 삭제는 제공하지 않는다**(`articles` FK 전부 `ON DELETE CASCADE`로 집계 소스 비가역 연쇄 삭제 위험 → 008 범위 밖, 후속 피처).
- **FR-035**: **숨김 일관성** — hidden 기사는 사용자 피드(003)·검색(003)·트렌드 집계 입력(007 키워드 추출 대상)·북마크 노출·알림 등 **모든 사용자향 읽기 경로에서 제외**되어야 하고, 일반 사용자의 **기사 상세/딥링크 접근은 404**(존재 비노출)여야 한다. **어드민 뷰에서는 hidden 포함 조회·상세가 가능**해야 한다.

**US4 — 공지 + 어드민 푸시**

- **FR-040**: 관리자는 공지(Notice)를 생성·수정·삭제하고 게시/비게시 상태를 전환할 수 있어야 한다(CRUD).
- **FR-041**: 시스템은 게시 상태인 공지만 공개 조회에 노출하고 초안/비게시 공지는 제외해야 한다.
- **FR-042**: 관리자는 005 알림 파이프라인을 통해 어드민 푸시/인앱 알림을 발송할 수 있어야 하며, 발송은 중복 없이(멱등) 처리되어야 한다.
- **FR-043**: 공지 생성/수정 시 제목·본문에 대한 기본 검증(빈 값·최대 길이)을 적용해야 한다.

**US5 — 심층 통계·에러 로그·수집량 상세**

- **FR-050**: 시스템은 기간별 운영 통계(OpsStats: 일자별 수집·요약·편향·트렌드 처리량 추이)를 제공해야 한다.
- **FR-051**: 시스템은 처리 실패 이벤트를 **기존 상태 데이터의 집계**(요약 `summary_status` FAILED·편향 status FAILED·알림 아웃박스 FAILED)로 조회(필터·페이지네이션·시간 역순) 제공해야 한다. 전용 에러 캡처 저장소(신규 테이블)는 도입하지 않는다.
- **FR-052**: 시스템은 소스별 수집량 상세(일자별 호출 수·예산 대비 사용률 등)를 드릴다운 제공해야 한다.

**감사(횡단)**

- **FR-060**: 시스템은 **변형(부수효과) 어드민 행위** — 역할 변경·계정 활성/비활성·기사 숨김/해제·스케줄러 제어(토글/수동 실행)·공지 CRUD·푸시 발송 — 를 `AdminAuditLog`에 감사 기록해야 한다(행위자·행위 유형·대상 유형/식별자·시각). **단순 조회는 감사 대상이 아니다.**

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자는 특정 사용자를 찾아 역할 또는 상태를 변경하는 작업을 30초 이내에 완료할 수 있다.
- **SC-002**: 비ADMIN(USER·비인증)의 모든 `/api/v1/admin/**` 호출은 100% 차단된다(401/403).
- **SC-003**: 운영 모니터링 대시보드는 단일 화면 진입에 필요한 핵심 지표(KPI·파이프라인·스케줄러 상태)를 5초 이내에 반환한다.
- **SC-004**: 자기 자신 강등/비활성화 및 마지막 관리자 lockout 시도는 100% 차단된다(운영자 0명 상태가 발생하지 않는다).
- **SC-005**: 관리자가 기사를 숨김 처리하면 그 즉시(다음 조회부터) 모든 사용자향 경로(피드·검색·트렌드)에서 노출 0건이 된다.
- **SC-006**: 게시 상태 변경(게시/비게시)은 공개 공지 조회에 즉시 반영된다(미게시 공지 노출 0건).
- **SC-007**: 데이터가 없는 신규 환경에서도 모든 모니터링·통계 조회가 오류 없이(0·빈 값) 응답한다.
- **SC-008**: 어드민 푸시 발송은 동일 발송 요청에 대해 중복 알림을 만들지 않는다(중복률 0%).
- **SC-009**: 모든 변형 어드민 액션(역할 변경·활성화·숨김·스케줄러 제어·공지 CRUD·푸시 발송)은 100% AdminAuditLog에 기록된다.
- **SC-010**: 관리자가 비활성화한 스케줄러는 앱 재기동 후에도 비활성 상태를 유지한다(설정 영속).

## Assumptions

- **인가 재활용**: 002의 JWT 인증 + 기존 `SecurityConfig`의 `/api/v1/admin/** → hasRole(ADMIN)` 규칙을 그대로 사용하며, 신규 역할(EDITOR 등)을 도입하지 않는다(USER/ADMIN 2값 유지).
- **읽기 전용 우선**: 모니터링(US2)·심층 통계(US5)는 기존 테이블 집계만으로 산출한다. **에러 로그는 기존 FAILED 상태(summary_status·bias status·알림 아웃박스)의 집계**이며 전용 에러 캡처 저장소를 신설하지 않는다.
- **수집 소스 계측 재활용**: 외부 수집 호출량은 기존 `source_daily_usage`(call_count)를 노출한다. AI/TTS/번역(Gemini·Polly·DeepL)의 호출·토큰·비용 계측은 **부재**하며 008 범위 밖(후속 ApiUsage 피처).
- **행동 데이터 부재**: 조회·완료율·AI클릭·공유 등 사용자 행동 분석(W-A-05)은 행동 이벤트 수집 인프라(009-event-tracking 선행 필요)가 없어 008 범위 밖. UserStats는 기존 데이터로 산출 가능한 항목에 한정한다.
- **단일 인스턴스 스케줄러**: 스케줄러는 단일 인스턴스 fixedDelay 전제(007과 동일). 수동 실행·토글은 이 가정 위에서 동작하며, enabled 토글은 `SchedulerSetting`으로 DB 영속(재기동 후 유지). 주기 동적 변경은 후속. 다중 인스턴스는 범위 밖.
- **가역 숨김만, 비가역 삭제 없음**: 기사 제어는 가역적 숨김(hidden)만 제공하고 영구 삭제는 없다 → `articles` CASCADE FK로 인한 집계(트렌드·편향) 무결성을 보존한다. 영구 삭제(법적/저작권 takedown)는 가드·감사·CASCADE 사전표시 설계 후 후속 피처.
- **UI 범위 밖**: 어드민 화면 렌더링(클라이언트 프론트엔드)은 범위 밖. 본 스펙은 어드민 API 계약만 정의한다.
- **의존 피처 구현 완료**: 001(수집)·002(인증/인가)·005(알림)·006(편향)·007(트렌드)이 모두 구현·머지된 상태를 전제한다.
