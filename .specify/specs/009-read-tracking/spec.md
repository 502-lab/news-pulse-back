# Feature Specification: 009 읽기 추적(Read Tracking) — 조회 이벤트 BE 단독 기록

**Feature Branch**: `feat/009-read-tracking`

**Created**: 2026-06-24

**Status**: Draft

**Input**: User description: "009-read-tracking 시작. scope = P1(조회 BE 단독)만. … 단일 article_event 테이블(안 A). 개인 읽은수/조회이력 조회 API. 체류·완료율·클릭·공유는 클라 계약 의존 → 후속 사이클 제외."

## 개요

기사 조회(읽음) 행동을 **백엔드 단독**으로 기록해, 개인화 인사이트(009 인사이트)와 편향 조회이력(006)이 요구하는 **개인 조회 이력**의 데이터 기반을 마련한다. 본 스펙의 구현 범위는 **P1 — 인증 사용자의 기사 상세 조회를 서버가 자동 기록하고, 사용자가 자신의 읽은 기사 수·조회 이력을 조회**하는 것까지다. 체류시간·완료율·AI클릭·공유 등 **클라이언트 계측이 필요한 이벤트는 클라이언트–BE 이벤트 계약에 의존하므로 후속 사이클(P2/P3)로 분리**하며, 본 스펙에서는 빈 구현(받을 준비만)을 만들지 않는다.

## Clarifications

### Session 2026-06-24

- Q: D1 — "읽은수" 정의와 중복 조회 처리 → A: **읽은수 = 고유 기사 수(distinct article)**. 이벤트는 매 조회 기록하되, **동일 기사 30분 내 재조회는 1건으로 디바운스**. "열람 횟수"는 필요 시 이벤트 행에서 별도 산출 가능하게 유지(읽은수 기본 지표는 distinct).
- Q: D2 — 조회 기록 동기/비동기 + 실패 격리 → A: **동기·best-effort**. 같은 요청에서 INSERT하되 **기록 실패(예외)는 try-catch로 삼켜 기사 상세 조회를 절대 깨지 않음**(실패 격리 — 보장). **기록 지연은 동기라 상세 응답에 전파될 수 있고 MVP가 수용**(지연 비격리 — try-catch는 예외만 잡고 타임아웃은 못 잡음). 저볼륨 동기 시작, **진짜 지연 격리(@Async 별도 스레드풀)는 forward-note**. `article_event`가 가장 빨리 성장하므로 인덱스(account_id, article_id, occurred_at) + 성장/파티셔닝 forward-note.
- Q: D3 — 비로그인(미인증) 조회 처리 → A: **미기록(인증 사용자만 기록)**. `account_id` NOT NULL. 익명 집계는 후속(필요 시).
- Q: D4 — 체류시간·완료율 정의 → A: **후속 사이클 deferral 확정**. 완료 임계 스크롤%·체류 측정 방식은 클라이언트 이벤트 계약 사이클에서 정의. 본 스펙은 deferral 명시만 유지.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 인증 사용자의 기사 조회 자동 기록 (Priority: P1)

로그인한 사용자가 기사 상세를 열면, 시스템이 별도의 클라이언트 동작 없이 "누가·어떤 기사를·언제 읽었는지"를 자동으로 기록한다. 사용자는 추가 행동을 하지 않으며, 클라이언트 앱도 변경되지 않는다(기존 상세 조회 호출 그대로).

**Why this priority**: 모든 후속 개인화(읽은수·조회이력·인사이트·개인 편향 집계)의 **원천 데이터**다. 이 기록이 없으면 하위 피처가 성립하지 않는다. 클라이언트 의존이 없어 단독으로 즉시 가치를 만든다.

**Independent Test**: 인증 사용자가 기사 상세를 1회 조회한 뒤, 해당 사용자·기사에 대한 조회 이벤트가 1건 생성됨을 저장소에서 확인. 비인증 호출/존재하지 않는 기사 조회 시의 처리도 검증 가능.

**Acceptance Scenarios**:

1. **Given** 로그인 사용자, **When** 기사 상세를 조회, **Then** (사용자, 기사, 조회시각) 조회 이벤트가 1건 기록되고 상세 응답은 정상 반환된다.
2. **Given** 로그인 사용자가 같은 기사를 30분 내 여러 번 조회, **When** 재조회, **Then** 디바운스(D1)로 중복 기록이 억제된다(30분 윈도우 내 1건).
3. **Given** 조회 기록 처리가 일시적으로 실패, **When** 기사 상세를 조회, **Then** 상세 응답은 영향 없이 정상 반환된다(기록은 best-effort, 핫패스 비차단).
4. **Given** 비인증(비로그인) 요청, **When** 기사 상세를 조회, **Then** 조회 이벤트는 기록되지 않는다(D3 — 인증만 기록).

### User Story 2 - 개인 읽은 기사 수·조회 이력 조회 (Priority: P2)

사용자가 자신이 지금까지 읽은 기사 수(읽은수)와 최근 조회한 기사 목록(조회 이력, 최신순)을 확인한다. 006(개인 편향 조회이력)·009 인사이트의 "읽은수"가 이 데이터를 소비한다.

**Why this priority**: US1이 만든 데이터를 사용자·하위 피처에 노출하는 소비 경로. US1 없이는 성립하지 않아 후순위(P2)지만 본 사이클에 포함한다("전부 P1 금지" 규칙 준수 + 빌드 순서 반영).

**Independent Test**: US1으로 N건의 조회를 만든 뒤, 본인 인증 토큰으로 읽은수=고유 기사 수, 조회 이력=시각 역순 목록이 반환되고, 타인 토큰으로는 본인 데이터에 접근할 수 없음을 확인.

**Acceptance Scenarios**:

1. **Given** 사용자가 서로 다른 기사 3건을 조회, **When** 읽은수 조회, **Then** 읽은수=고유 기사 3(distinct, D1)이 반환된다.
2. **Given** 사용자가 기사들을 조회, **When** 조회 이력 조회, **Then** 최근 조회 순으로 기사 목록이 페이지네이션되어 반환된다.
3. **Given** 사용자 A의 토큰, **When** 사용자 B의 읽은수/이력 접근 시도, **Then** 거부된다(본인 데이터만).

### User Story 3 - [후속 사이클 — 본 스펙 구현 제외] 클라이언트 계측 이벤트 (Priority: P3)

체류시간·완료율(스크롤%)·AI클릭·공유는 클라이언트가 측정·전송해야 하는 이벤트다. 본 스펙은 **구현하지 않으며**, 별도 사이클(read-tracking P2 / 009 인사이트 P2)에서 클라이언트–BE 이벤트 계약(완료 임계 스크롤%·체류 측정 방식·배치 전송·검증)과 함께 다룬다.

**Why this priority**: 클라이언트 파트 의존(BE 단독 불가). 계약 미확정 상태에서 BE에 수신 엔드포인트/스키마를 미리 만들면 빈 구현·거짓 약속이 된다. 단, 본 스펙의 저장소 스키마는 후속 확장을 막지 않는 **forward-seam**(아래 Key Entities의 event_type/metric_value/source)을 남긴다.

**Independent Test**: 해당 없음(본 스펙 미구현). 후속 사이클에서 정의.

---

### Edge Cases

- 존재하지 않거나 숨김(admin_hidden) 기사를 조회: 상세 조회 자체가 404(008 #9 가드)이면 조회 이벤트도 기록되지 않는다.
- 동일 사용자·동일 기사 단시간 다중 조회: 디바운스(D1)로 이력 폭증 방지.
- 조회 저장소 쓰기 지연/실패: 상세 응답을 막지 않는다(best-effort, 비차단).
- 계정 삭제/비활성: 조회 이벤트의 보존·정리 정책은 Assumptions(개인정보·보존) 참조.
- 대량 조회로 인한 테이블 성장: 인덱스·보존 정책 forward-note(plan에서 구체화).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 인증 사용자가 기사 상세를 조회할 때 (사용자 식별자, 기사 식별자, 조회 시각)로 구성된 조회 이벤트를 기록해야 한다.
- **FR-002**: 조회 기록은 **기존 기사 상세 조회 경로에서 서버 측으로** 수행되어야 하며, 클라이언트의 추가 호출·변경을 요구하지 않아야 한다.
- **FR-003**: 조회 기록은 **동기·best-effort**여야 한다 — 기사 상세 조회와 동일 요청에서 기록하되, **기록 실패(예외: DB 오류·제약 위반 등)는 삼켜(try-catch) 기사 상세 응답의 성공 여부에 영향을 주지 않아야 한다**(실패 격리, 핫패스 보호). **단, 기록의 지연(슬로우쿼리·DB 일시지연)은 동기 호출 특성상 상세 응답 시간에 전파될 수 있으며 MVP는 이를 수용한다**(저볼륨+디바운스로 대부분 SELECT-only). 진짜 지연 격리(별도 스레드풀 비동기)는 forward-note(볼륨 증가 시).
  - **best-effort 정의**: try-catch가 잡는 것 = **예외(O)** → 실패 비차단(보장). 못 잡는 것 = **타임아웃·지연(X)** → 지연 비격리(MVP 수용).
- **FR-004**: 시스템은 조회 이벤트를 **단일 이벤트 저장소**에 저장하며, 이벤트 유형·출처를 구분 가능한 형태로 보관해야 한다. **본 스펙(P1)에서는 유형=조회(VIEW)·출처=서버(SERVER)만 생성·사용**한다.
- **FR-005**: 저장소는 후속(체류·완료율·클릭·공유) 확장을 위한 **forward-seam**(이벤트 유형 구분, 수치 측정값 보관 필드, 출처 구분)을 가지되, 본 스펙에서는 VIEW·SERVER 외 값을 기록하지 않아야 한다(빈 구현 금지).
- **FR-006**: 사용자는 **자신이 읽은 기사 수(읽은수)**를 조회할 수 있어야 한다. **읽은수 = 고유 기사 수(distinct article)**로 집계한다(열람 횟수는 별도 산출 가능하되 기본 지표 아님).
- **FR-007**: 사용자는 **자신의 조회 이력(읽은 기사 목록)**을 최신 조회순으로 조회할 수 있어야 하며, 페이지네이션을 지원해야 한다.
- **FR-008**: 읽은수·조회 이력 조회는 **인증된 본인만** 접근 가능해야 한다(타인 데이터 접근 불가).
- **FR-009**: 동일 사용자·동일 기사의 **30분 내 재조회는 1건으로 디바운스**해야 한다(이력 폭증 방지). 매 조회는 기록 시도하되 30분 윈도우 내 중복은 억제한다.
- **FR-010**: 조회 이벤트 데이터는 **개인화 인사이트(009 인사이트)와 개인 편향 조회이력(006)**이 소비할 수 있는 형태로 제공되어야 한다.
- **FR-011**: 비인증(비로그인) 사용자의 기사 조회는 **기록하지 않는다(인증 사용자만)**. `account_id`는 NOT NULL이며, 익명 집계는 후속 범위.

### Key Entities *(include if feature involves data)*

- **ArticleEvent (조회 이벤트, 신규 — 단일 테이블 "안 A")**: 사용자의 기사 행동 이벤트 1건. 주요 속성:
  - **account_id**: 행동 주체(인증 사용자). **NOT NULL**(비로그인 미기록 — D3).
  - **article_id**: 대상 기사(003 `articles` 참조).
  - **event_type**: 이벤트 유형. **P1=VIEW만 사용**. (forward-seam: 후속 DWELL/COMPLETE/AI_CLICK/SHARE)
  - **metric_value**: 수치 측정값(체류ms·완료율 등). **P1에서는 null**. (forward-seam)
  - **source**: 이벤트 출처. **P1=SERVER만 사용**(서버 기록 — 클라 위조 방지). (forward-seam: CLIENT)
  - **occurred_at**: 발생 시각.
  - 소비처: 읽은수=account별 고유 article 집계(distinct), 조회 이력=account별 occurred_at 역순.
  - **인덱스/성장 forward-note**: 가장 빠르게 성장하는 테이블 → 인덱스(account_id, article_id, occurred_at). 디바운스(30분 윈도우)·읽은수(distinct)·이력(역순) 질의를 지원. 대량 성장 시 파티셔닝(occurred_at 기준)·보존 정책은 plan에서 구체화.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 인증 사용자의 기사 상세 조회 1건당 조회 이벤트가 **정확히 정책에 맞게(디바운스 반영)** 1건 기록된다(누락 0, 중복은 디바운스 범위 내 0).
- **SC-002**: 조회 기록의 **실패는 기사 상세 조회 성공을 절대 깨지 않는다**(실패 격리 — 보장). 기록이 핫패스에 더하는 비용은 **디바운스 SELECT 1 + 조건부 INSERT 1**로 한정되며, 지연 전파는 동기 호출 특성상 존재하나 저볼륨+디바운스로 작게 유지된다(완전 지연 격리는 forward-note의 비동기로 달성).
- **SC-003**: 사용자는 자신의 읽은수·조회 이력을 **본인 인증만으로** 정확히 조회할 수 있고, 타인 데이터는 **0건** 노출된다.
- **SC-004**: 조회 이벤트가 009 인사이트·006 개인 편향 집계의 입력으로 **사용 가능**(개인 조회 이력 질의가 성립)하다.
- **SC-005**: 저장소 스키마가 후속 클라이언트 이벤트(체류·완료율·클릭·공유)를 **스키마 변경 없이 수용**할 수 있다(forward-seam 검증).

## Resolved Decisions (clarify 2026-06-24 확정)

- **D1 — "읽은수" 정의 + 디바운스**: **확정** — 읽은수 = **고유 기사 수(distinct article)**. 이벤트는 매 조회 기록하되 **동일 기사 30분 내 재조회는 1건 디바운스**. 열람 횟수는 이벤트 행에서 별도 산출 가능(기본 지표는 distinct). → FR-006·FR-009.
- **D2 — 조회 기록 동기/비동기**: **확정** — **동기·비차단(best-effort)**. 동일 요청 INSERT, 기록 실패는 try-catch로 삼켜 상세 조회 비차단. 비동기(큐/배치)는 forward-note. → FR-003 + Key Entities 인덱스/파티셔닝 forward-note.
- **D3 — 비로그인 조회 처리**: **확정** — **미기록(인증만)**, `account_id` NOT NULL. 익명 집계는 후속. → FR-011.
- **D4 — 체류시간·완료율 정의**: **deferral 확정** — 후속 클라이언트 계약 사이클에서 정의(완료 임계 스크롤%·체류 측정 방식). 본 스펙 미정의. → Out of Scope.

## Assumptions

- 기사 상세 조회 경로(`기사 상세` 엔드포인트)는 **인증된 요청**으로 처리되어 행동 주체(account)를 서버가 식별할 수 있다(002 인증 재사용). 비로그인 노출 경로(공개 피드/트렌드)는 본 스펙의 조회 기록 대상이 아니다.
- 신규 저장소(ArticleEvent) 외 기존 테이블 변경은 없다(003 `articles`·002 `accounts` 참조만).
- 조회 이벤트는 개인 데이터다 — 보존·삭제(계정 파기 시)는 기존 개인정보 정책을 따른다(세부는 plan).
- 읽은수·조회 이력 조회 API는 인증 사용자 본인 스코프(`/api/v1/me/...` 류)로 노출한다.
- 클라이언트 계측·UI 렌더링은 범위 밖이다.

## Dependencies

- **003 (개인화 피드·검색·저장)**: `articles` 도메인 및 기사 상세 조회 경로.
- **002 (계정·인증·온보딩)**: 인증 주체(account) 식별, 본인 스코프 접근 제어.
- **006 (편향 분석)**: 개인 편향 조회이력의 **소비처**(006 spec: "개인 조회 이력 기반은 009 인사이트에서 담당").
- **009 인사이트(후속)**: 본 스펙의 조회 이벤트를 "읽은수" 입력으로 소비. "평균읽기시간"은 체류(P2) 의존이라 본 스펙 범위 밖.

## Out of Scope (후속 사이클)

- 체류시간·완료율(스크롤%)·AI클릭·공유 이벤트 및 이를 위한 **클라이언트–BE 이벤트 수신 엔드포인트/계약** → read-tracking P2 별도 사이클.
- 009 인사이트의 "평균읽기시간"(체류 의존).
- 클라이언트 계측(스크롤·포커스·타이머)·UI.
- 온보딩 age/job 저장 점검(별도 트랙 — 본 스펙과 무관, 섞지 않음).
